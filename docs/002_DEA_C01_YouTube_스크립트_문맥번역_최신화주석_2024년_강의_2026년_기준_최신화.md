# 1. 공통 사전 설정

## 1.1 챕터 2. Setup Work 02:48–11:56

이제 뒤의 실습에서 계속 재사용할 공통 AWS 리소스를 먼저 만듭니다. GitHub 저장소를 로컬에 내려받아 압축을 풀어 두어야 합니다. 이후 AWS에 업로드할 코드와 데이터가 이 저장소에 들어 있기 때문입니다.

CloudFormation으로 다음 리소스를 생성합니다.

- VPC
- 퍼블릭 서브넷 2개
- 보안 그룹
- 라우팅 관련 리소스
- S3 버킷

S3 버킷은 강의 전반에서 데이터를 저장하는 공용 작업 공간 역할을 합니다. CloudFormation은 인프라를 코드로 선언해 리소스를 만드는 AWS 서비스입니다. 이 강의에서는 이미 준비된 YAML 템플릿을 실행해 기본 환경을 빠르게 만드는 데 사용합니다.

GitHub의 `setup-code` 폴더 안 `setup-code.yaml`을 CloudFormation에 업로드합니다.

1. AWS 콘솔에서 **CloudFormation**으로 이동합니다.
2. **Create stack**을 선택합니다.
3. 새 리소스 생성과 기존 템플릿 사용을 선택합니다.
4. 로컬 파일 업로드를 선택한 뒤 `setup-code.yaml`을 지정합니다.
5. 스택 이름은 원하는 이름으로 정합니다.
6. 환경 이름은 예시로 `dev`를 사용합니다.
7. 퍼블릭 서브넷 관련 입력값은 무엇을 의미하는지 모르면 기본값을 유지합니다.
8. S3 버킷 이름을 입력합니다.

S3 버킷 이름은 AWS 전체에서 전역적으로 고유해야 합니다. 이미 다른 사람이 사용 중인 이름이면 CloudFormation 스택 생성이 실패합니다. 따라서 본인 식별자와 임의의 숫자·문자를 조합해 중복 가능성을 낮춥니다.

이 강의에서는 콘솔에 로그인한 역할이 관리자 권한을 갖는다는 전제입니다. 개인 AWS 계정의 루트 사용자라면 대부분의 권한을 갖고 있을 수 있지만, 실무 환경에서는 별도 권한이 필요할 수 있습니다.

스택을 제출한 뒤 생성 상태를 확인합니다. VPC까지 만들기 때문에 몇 분에서 길게는 수십 분이 걸릴 수 있습니다. `CREATE_COMPLETE`가 녹색으로 표시되면 완료입니다. 실패 시에는 이벤트 탭을 보고 원인을 확인합니다. 흔한 원인은 S3 버킷 이름 중복입니다.

스택이 완료된 뒤 **Resources**에서 생성된 리소스를 확인하고, **Outputs**에서 이후에 필요한 값도 확인합니다.

다음으로 S3 버킷 안에 강의용 디렉터리 구조를 만듭니다.

```text
s3://<your-bucket>/
├── raw-data/
├── process-data/
├── script-location/
├── temp-directory/
└── athena/
```

- `raw-data`: 원본 데이터를 두는 위치
- `process-data`: 변환된 데이터를 두는 위치
- `script-location`: Glue, EMR 등의 스크립트를 두는 위치
- `temp-directory`: 임시 파일, 로그, 중간 산출물 위치
- `athena`: Athena 쿼리 결과 위치

이후 `raw-data` 아래에 `customers`, `employees`, `orders` 데이터를 업로드합니다. 각 폴더에는 CSV 파일이 들어갑니다. 여기까지가 공통 환경 설정입니다. 이후 강의에서는 이 버킷, VPC, 서브넷을 반복 사용합니다.

강사는 이 강의가 시험 전 범위를 완전히 포괄하지 않는다고 다시 강조합니다. AWS 경험이 부족하다면 이 영상만 보지 말고 서비스별 공식 문서와 별도 자료를 통해 보완해야 합니다.

# 2. 데이터 엔지니어링 기초

## 2.1 챕터 3. Data Engineering Fundamentals 11:57–12:20

이제부터 슬라이드와 실습을 섞어 진행합니다. 슬라이드보다 상세한 보충 복습 자료가 있을 수 있으나, 강의의 본문만으로도 기본 구조는 이해할 수 있습니다.

AWS Data Engineer Associate 시험을 준비할 때는 먼저 데이터의 유형과 성질, 저장·처리 구조, 데이터 모델링, 품질, SQL 같은 기초를 이해해야 합니다.

## 2.2 챕터 4. Types of Data 12:21–15:27

### 정형 데이터 Structured Data

정형 데이터는 미리 정의된 구조에 따라 조직된 데이터입니다. 일반적으로 행과 열로 이루어진 표 형태이며, 필드 이름과 데이터 타입이 분명합니다. 전통적인 SQL 질의로 비교적 쉽게 검색하고 분석할 수 있습니다.

예시는 다음과 같습니다.

- MySQL, PostgreSQL 같은 관계형 데이터베이스
- Amazon Redshift 같은 데이터 웨어하우스
- 행·열·헤더 구조를 가진 스프레드시트

### 비정형 데이터 Unstructured Data

비정형 데이터는 미리 정해진 표 구조나 고정 스키마가 없습니다. 전통적인 관계형 방식만으로는 검색·분석이 어렵고, 데이터 형식도 매우 다양합니다.

예시는 다음과 같습니다.

- 긴 자유 형식의 텍스트 문서
- 이미지, 음성, 동영상 같은 멀티미디어 파일
- 소셜 미디어 게시물
- 사람이 자유로운 문체로 작성한 이메일 본문

예를 들어 음성 녹음은 파형 자체에 열과 행이 있는 것이 아닙니다. 음성을 텍스트로 전사하거나 특징을 추출한 뒤에야 분석에 쓰기 좋은 형태가 됩니다.

### 반정형 데이터 Semi-structured Data

반정형 데이터는 정형과 비정형 사이의 성격을 갖습니다. 엑셀처럼 완전히 고정된 표 구조는 아니지만, 태그·키·마커 같은 조직화 단서가 존재합니다.

예시는 다음과 같습니다.

- XML: 태그로 요소의 구조를 표현함
- JSON: 키와 값의 쌍으로 구조를 표현함
- DynamoDB 같은 NoSQL 데이터
- HTML: `head`, `body` 같은 태그는 구조를 제공하지만, 내부 본문은 자유 텍스트일 수 있음

핵심은 데이터 전체가 완전히 고정된 행·열은 아니더라도, 파싱할 수 있는 규칙이나 구조적 표식이 존재한다는 점입니다.

## 2.3 챕터 5. Properties of Data 15:28–19:53

데이터 엔지니어링에서는 흔히 데이터의 세 가지 V를 봅니다.

- **Volume**: 데이터의 양
- **Velocity**: 데이터가 생성·전송·처리되는 속도
- **Variety**: 데이터의 종류, 형식, 출처의 다양성

### Volume 데이터 규모

Volume은 생성·저장·처리해야 하는 데이터의 양입니다. MB 수준의 작은 데이터부터 TB, PB 규모의 데이터까지 포함합니다.

데이터 규모가 커질수록 다음 문제가 생깁니다.

- 저장소를 어떻게 확장할 것인가
- 처리에 필요한 컴퓨팅 자원을 어떻게 확보할 것인가
- 대규모 데이터를 여러 원천에서 어떻게 통합할 것인가
- 데이터 거버넌스와 카탈로그를 어떻게 운영할 것인가

AWS 시험 문제에서는 데이터 양이 주어지고, 그 규모를 비용·성능 측면에서 감당할 서비스가 무엇인지 묻는 식으로 나옵니다.

예를 들면 다음처럼 생각합니다.

- 대규모 데이터 저장: Amazon S3, Amazon Redshift
- 관계형 트랜잭션 데이터: Amazon RDS
- 대규모 분산 처리: Amazon EMR과 Apache Spark
- 서버 관리 없이 ETL 처리: AWS Glue
- 데이터의 위치·스키마·소유 정보를 관리: AWS Glue Data Catalog
- 데이터 공유와 거버넌스: AWS DataZone, Lake Formation 등

### Velocity 데이터 속도

Velocity는 데이터가 생성되고, 전송되고, 처리되는 속도입니다.

시험 문제를 읽을 때는 아래 두 질문부터 해야 합니다.

1. 실시간 또는 준실시간 처리가 필요한가
2. 허용 가능한 지연 시간 latency은 얼마인가

데이터가 도착하자마자 처리되어야 한다면 스트리밍·메시징 계열 서비스를 검토합니다. 하루 뒤 처리해도 된다면 비용 효율적인 배치 처리도 가능합니다.

즉, 낮은 지연 시간은 일반적으로 더 높은 처리 능력과 비용을 요구하고, 지연을 허용하면 배치 처리로 비용을 낮출 여지가 생깁니다.

### Variety 데이터 다양성

Variety는 데이터의 출처, 유형, 표현 형식이 얼마나 다양한지를 뜻합니다. CSV, JSON, 로그, 관계형 테이블, API 응답, 이미지·문서 등이 한 시스템 안에 섞일 수 있습니다.

다양성이 커질수록 발생하는 문제는 다음과 같습니다.

- 서로 다른 형식의 데이터를 함께 통합하기 어려움
- 데이터 품질 기준이 달라짐
- 서비스별 데이터 표현 차이 때문에 상호운용성이 낮아짐
- 공통 스키마·카탈로그·변환 규칙이 필요해짐

AWS Glue는 여러 형식의 데이터를 읽고 변환하는 데 쓰일 수 있습니다. 다만 서비스 이름만 외우지 말고, 데이터의 양·속도·다양성 중 무엇이 문제인지 먼저 판단해야 합니다.

## 2.4 챕터 6. Data Warehouse vs Data Lake vs Data Lakehouse 19:54–25:16

### 데이터 웨어하우스 Data Warehouse

데이터 웨어하우스는 여러 원천의 **정형 데이터**를 중앙에 모아 BI, 분석, 리포팅을 지원하는 저장소입니다.

핵심 특징은 **Schema-on-write**입니다. 데이터를 웨어하우스에 넣기 전에 정제하고 변환하고, 정해진 스키마를 충족하는지 검증합니다. 따라서 품질과 일관성이 높은 분석용 테이블을 만들기 좋습니다.

보통 다음 목적에 적합합니다.

- 빠른 분석 쿼리
- 반복적인 BI 리포트
- 강한 스키마 통제
- 데이터 무결성과 일관성이 중요한 분석

예시는 Amazon Redshift, Snowflake, Google BigQuery입니다.

### 데이터 레이크 Data Lake

데이터 레이크는 정형·반정형·비정형 데이터를 원본 형식에 가깝게, 매우 큰 규모로 저장하는 중앙 저장소입니다.

핵심 특징은 **Schema-on-read**입니다. 데이터를 넣을 때 모든 스키마를 강하게 고정하지 않고, 읽고 분석할 때 필요한 해석과 스키마를 적용합니다.

일반적인 특성은 다음과 같습니다.

- 원시 데이터 보존
- 구조화 여부와 무관한 다양한 형식 저장
- 컴퓨팅과 저장소의 분리
- 대규모 확장성
- 데이터 탐색과 실험에 적합

AWS에서는 보통 Amazon S3가 데이터 레이크 저장소 역할을 합니다.

### 데이터 레이크하우스 Data Lakehouse

데이터 레이크하우스는 데이터 레이크의 유연성과 웨어하우스의 관리·분석 특성을 결합하려는 접근입니다.

목표는 다음과 같습니다.

- 정형·반정형·비정형 데이터를 함께 처리
- Schema-on-read와 Schema-on-write를 모두 활용
- 분석과 일부 실시간 처리 지원
- 데이터 복제와 시스템 간 이동을 줄임
- 레이크 위에서 더 강한 테이블 관리와 거버넌스를 제공

강의에서는 Databricks와 Google BigLake를 예시로 듭니다.

시험에서의 단순 판단 기준은 다음과 같습니다.

| 상황 | 우선 떠올릴 구조 |
|---|---|
| 정형 데이터, 강한 스키마 통제, 빠른 BI 분석 | 데이터 웨어하우스 |
| 매우 다양한 원본 데이터, 저비용 대규모 저장, 탐색 | 데이터 레이크 |
| 레이크 유연성 + 테이블 관리·분석 성능·거버넌스 필요 | 데이터 레이크하우스 |

비용 문제에서 비정형 원시 데이터를 매우 저렴하게 대량 저장해야 한다면 데이터 레이크를 먼저 생각합니다. 반대로 고성능 분석·관리 특성이 더 중요하다면 웨어하우스나 레이크하우스 성격을 검토합니다.

## 2.5 챕터 7. Data Mesh 25:17–30:35

데이터 메시 Data Mesh는 중앙 데이터 팀 하나가 모든 데이터를 수집·변환·배포하는 구조의 병목을 줄이려는 조직·아키텍처 접근입니다.

전통 구조에서는 파일, PostgreSQL, 외부 API 같은 여러 데이터 원천을 중앙 데이터 엔지니어링 팀이 ETL로 중앙 저장소에 모읍니다. 비즈니스 부서는 그 팀에 요청을 넣고, 데이터 팀은 백로그에 따라 처리합니다.

이 구조의 문제는 다음과 같습니다.

- 데이터 지식이 중앙 팀에 과도하게 집중됨
- 변경 요청이 중앙 팀의 백로그에서 오래 대기함
- 데이터 엔지니어가 도메인 의미를 완전히 알기 어려움
- 데이터 생산자와 소비자 사이의 거리 때문에 정확도가 떨어질 수 있음

데이터 메시는 데이터의 관리와 접근을 도메인별 팀으로 분산합니다.

예를 들어 음악 서비스라면 다음처럼 나눌 수 있습니다.

- 아티스트 도메인 팀: 아티스트 데이터 제품을 책임짐
- 곡 도메인 팀: 곡 데이터 제품을 책임짐
- 다른 팀: 두 데이터 제품을 조합해 새로운 데이터 제품을 만듦

아래 계층에는 저장소, 파이프라인, ETL, 접근 제어, 카탈로그 같은 공통 플랫폼이 있습니다. 각 도메인 팀은 이를 셀프서비스 방식으로 활용합니다. 상위에는 모든 팀이 공유할 이름 규칙, 보안, 상호운용성 기준을 정하는 연합형 거버넌스가 있습니다.

데이터 메시의 핵심 원칙은 다음과 같습니다.

1. 데이터 소유권은 데이터를 가장 잘 아는 도메인 팀에 둠
2. 데이터를 다른 팀이 소비할 수 있는 **제품**으로 다룸
3. 플랫폼은 셀프서비스로 제공함
4. 전사 표준은 연합형 의사결정으로 관리함

즉, 모든 것을 중앙화하지도 않고, 아무 기준 없이 각자도생하지도 않는 구조입니다.

## 2.6 챕터 8. Orchestration 30:36–31:33

오케스트레이션 Orchestration은 복잡한 데이터 워크플로우와 처리 과정을 자동으로 조정·관리하는 일입니다.

오케스트레이션 도구는 다음을 담당합니다.

- 작업 실행 시점 스케줄링
- 작업 간 의존성 관리
- 성공·실패 상태 감시
- 재시도와 오류 처리
- 여러 서비스의 실행 순서 조정

예를 들어 `S3 도착 → Glue Crawler → Glue ETL → 품질 검사 → Redshift 적재 → 알림` 같은 파이프라인을 사람이 매번 실행하지 않도록 만드는 역할입니다.

대표 예시는 Apache Airflow와 AWS Step Functions입니다.

## 2.7 챕터 8 후반. Common Data Formats and Data Sources 31:34–34:01

데이터 포맷은 시스템 사이에서 데이터를 어떤 구조로 저장하고 교환하는지를 뜻합니다.

- **CSV**: 쉼표로 값을 구분하는 텍스트 파일
- **JSON**: 키와 값 구조로 표현하는 형식
- **XML**: 태그로 구조를 표현하는 형식
- **Apache Parquet**: 대규모 분석에 자주 쓰는 컬럼형 파일 형식

데이터 원천 Data Source는 수집·분석·저장을 위해 데이터를 가져오는 출발점입니다.

예시는 다음과 같습니다.

- PostgreSQL, MySQL, Amazon RDS 같은 운영 데이터베이스
- Amazon Redshift 같은 데이터 웨어하우스
- Amazon S3 기반 데이터 레이크
- CSV 같은 평면 파일
- API와 외부 SaaS 애플리케이션

## 2.8 챕터 9. Data Modeling 34:02–36:01

데이터 모델링은 복잡한 시스템의 데이터 구조와 관계를 시각적·논리적으로 표현하는 작업입니다.

### 개념 모델 Conceptual Model

데이터가 무엇이고 어떤 큰 관계를 가지는지 블록 수준에서 표현합니다. 세부 컬럼이나 구현 방식은 아직 넣지 않습니다.

### 논리 모델 Logical Model

개념 모델에 엔터티, 속성, 관계, 키, 업무 규칙을 더합니다. 특정 DBMS 구현과는 분리되어 있지만, 데이터 구조는 구체화됩니다.

### 물리 모델 Physical Model

실제 DB나 분석 시스템에 구현할 테이블, 컬럼 타입, 인덱스, 파티션, 제약 조건 등을 표현합니다.

시험에서는 데이터 모델 유형에 맞는 저장소를 고르는 문제가 나올 수 있습니다.

- 키와 값 중심의 낮은 지연 시간 웹 워크로드: DynamoDB
- 관계형·트랜잭션 데이터: Amazon RDS 또는 Aurora
- 대규모 분석용 컬럼 저장과 BI: Amazon Redshift

핵심은 데이터 모델의 형태와 액세스 패턴을 읽고 서비스 선택으로 연결하는 것입니다.

## 2.9 챕터 9 후반. Data Lineage 36:02–37:15

데이터 계보 Data Lineage는 데이터가 조직 안에서 어디서 왔고, 어떤 변환을 거쳐, 어디로 이동했는지 추적하는 것입니다.

예를 들어 원천 A와 B가 있고, 이를 집계하고, B를 복제하고, 조인해 새 데이터셋을 만들었다면, 이 흐름과 변환 관계를 기록하는 것이 계보입니다.

데이터 계보가 필요한 이유는 다음과 같습니다.

- 오류가 발생한 데이터의 원인 추적
- 규제·감사 대응
- 변경 영향 분석
- 데이터 신뢰도 검증
- 누가 어떤 데이터를 사용했는지 파악

강의에서는 AWS Glue 작업과 연계할 수 있는 OpenLineage 계열 접근, 그리고 거버넌스·카탈로그·계보 기능을 제공하는 Collibra 같은 도구를 언급합니다.

## 2.10 챕터 10. Schema Evolution 37:16–37:46

스키마 진화 Schema Evolution는 시간이 지나면서 데이터베이스나 데이터 구조의 스키마가 바뀌는 것을 관리하는 과정입니다.

예를 들어 기존 테이블에 `first_name`, `last_name`만 있었는데 `full_name` 컬럼을 추가하려 한다면, 기존 데이터·ETL·다운스트림 쿼리·스키마 호환성까지 고려해 안전하게 변경해야 합니다.

단순히 컬럼 하나를 추가하는 문제가 아니라, 변경 뒤에도 파이프라인과 소비자가 계속 정상 동작하게 만드는 문제입니다.

## 2.11 챕터 11. Data Sampling 37:47–39:26

데이터 샘플링은 큰 데이터셋에서 일부만 선택해 분석하는 통계 기법입니다.

- **Population 모집단**: 전체 대상 데이터
- **Sample 표본**: 모집단에서 뽑은 일부 데이터
- **Sampling frame 표본틀**: 실제로 표본을 뽑을 수 있는 대상 목록 또는 범위
- **Sampling error 표본오차**: 표본 결과와 모집단 실제 특성의 차이
- **Sample size 표본 크기**: 뽑은 표본의 수

데이터 엔지니어는 통계 분석의 주 담당자는 아닐 수 있지만, 표본이 전체를 얼마나 잘 대표하는지, 표본 규모가 충분한지, 데이터가 편향되지 않았는지는 이해해야 합니다.

## 2.12 챕터 12. Data Skewness 39:27–40:23

데이터 왜도 Skewness는 데이터 값 분포가 평균을 기준으로 얼마나 비대칭인지를 뜻합니다.

- 왼쪽 꼬리가 긴 분포: 음의 왜도 negative skew
- 오른쪽 꼬리가 긴 분포: 양의 왜도 positive skew

시험에서는 그래프를 보고 왜도 방향을 식별하거나, 분포의 치우침이 분석·처리에 어떤 문제를 만들 수 있는지 묻는 정도를 예상할 수 있습니다.

[번역 주석] 여기서 말하는 통계적 왜도와, Spark·Redshift 등 분산 처리에서 특정 키에 데이터가 몰리는 **데이터 스큐 data skew**는 관련은 있지만 동일한 개념으로 취급하면 안 됩니다.

## 2.13 챕터 13. Data Validation and Profiling 40:24–40:56

데이터 검증 Data Validation은 데이터가 정확하고 일관되며, 미리 정한 규칙을 만족하는지 확인하는 일입니다.

데이터 프로파일링 Data Profiling은 데이터셋의 구조, 분포, 값의 범위, 누락치, 고유값 수, 품질 특성을 분석해 이해하는 일입니다.

간단히 구분하면 다음과 같습니다.

- 검증: 데이터가 기대한 규칙을 통과했는가
- 프로파일링: 데이터가 실제로 어떤 모습인가

## 2.14 챕터 14. A Word on SQL 40:57–41:39

이 자격증에서는 SQL을 알아야 합니다. 이후 Athena 파트에서 일부 SQL 문법을 보겠지만, 조인, 필터, 집계 같은 기본 SQL 자체를 처음부터 설명하지는 않습니다.

SQL이 낯설거나 오래전에 배웠다면 별도 복습이 필요합니다. AWS 서비스 선택 문제뿐 아니라, Athena·Redshift에서 데이터를 읽고 조작하는 기본 쿼리를 이해해야 하기 때문입니다.

# 3. AWS Glue

## 3.1 챕터 15. AWS Glue 41:40–44:23

AWS Glue는 AWS가 관리하는 완전관리형 ETL 서비스입니다. 사용자가 서버나 클러스터 인프라를 직접 프로비저닝하지 않아도 됩니다.

Glue의 핵심 요소는 다음과 같습니다.

- **AWS Glue Data Catalog**: 데이터의 위치, 스키마, 테이블, 연결 정보 같은 메타데이터를 관리하는 중앙 카탈로그
- **Glue Job**: 데이터를 추출·변환·적재하는 ETL 작업
- **ETL Script**: Job이 실행할 변환 로직
- **Trigger와 Schedule**: 시간 또는 이벤트 기반으로 Job을 실행하는 기능

Glue는 S3, 데이터베이스, 온프레미스 등에서 데이터를 읽어 변환한 뒤, 다른 S3 위치나 데이터 저장소로 적재할 수 있습니다.

시험 관점에서 `서버를 관리하지 않는 ETL 서비스`, `AWS에서 ETL 작업을 빠르게 구현`, `카탈로그 기반으로 데이터 위치와 스키마를 관리`라는 요구가 나오면 AWS Glue를 먼저 후보로 떠올립니다.

## 3.2 챕터 16. AWS Glue Hands-on Setup Work 44:24–50:23

Glue 실습에서는 앞서 만든 S3 버킷과 데이터를 재사용합니다.

GitHub의 `glue-code` 폴더에 있는 CloudFormation YAML 템플릿을 실행해 Glue 실습용 IAM 역할을 만듭니다. 이 역할은 Glue가 S3 데이터와 카탈로그에 접근할 수 있게 하는 권한을 제공합니다.

스택을 생성할 때 S3 버킷 ARN을 입력해야 합니다. 이는 IAM 정책이 특정 S3 버킷에 접근할 수 있도록 만들기 위해 필요합니다.

그 다음 Glue Crawler 실습을 위해 헤더가 포함된 데이터셋 두 개를 `raw-data` 위치에 업로드합니다.

- `employees_headers`
- `orders_headers`

Crawler는 CSV 헤더를 이용해 컬럼 이름을 추론할 수 있습니다. 원본 데이터에 헤더가 없으면 컬럼명까지 원하는 방식으로 자동 추론하기 어려우므로, 데모를 위해 헤더 포함 파일을 별도로 사용합니다.

S3 콘솔의 **Select** 기능을 이용하면 파일을 간단한 SQL로 미리 살펴볼 수 있습니다. 이 실습의 목적은 데이터가 예상대로 업로드됐는지 확인하는 것입니다.

## 3.3 챕터 17. AWS Glue Data Catalog 50:24–52:35

AWS Glue Data Catalog는 Glue의 핵심 구성 요소입니다. 쉽게 말해 **데이터 그 자체가 아니라, 데이터에 대한 설명서와 찾아가는 지도**입니다.

Data Catalog는 지속적으로 보관되는 메타데이터 저장소입니다. 데이터의 위치, 스키마, 데이터 타입, 분류 정보, 연결 정보 등을 저장·공유합니다.

메타데이터 예시는 다음과 같습니다.

- 데이터가 있는 S3 URI 또는 데이터베이스 위치
- 테이블과 컬럼 이름
- 컬럼 데이터 타입
- 파일 형식 CSV, Parquet 등
- 민감정보·PII 여부 같은 분류 정보
- 데이터 저장소에 접근하는 연결 정보

중요한 점은 Glue Catalog에 테이블을 등록해도 원본 데이터가 이동하지 않는다는 것입니다. S3의 파일은 S3에 그대로 있고, PostgreSQL의 테이블은 PostgreSQL에 그대로 있습니다. Catalog는 그 데이터를 어떻게 찾고 해석할지 기록합니다.

강의에서는 리전에 Glue Data Catalog가 하나 존재한다는 점, IAM 정책으로 접근을 통제한다는 점, 태그·민감도 정보와 결합해 데이터 거버넌스에도 활용할 수 있다는 점을 강조합니다.

## 3.4 챕터 18. AWS Glue Database 52:36–55:46

Glue Database는 실제 RDS 데이터베이스 인스턴스와 같은 의미가 아닙니다. Glue Data Catalog 안에서 테이블 정의를 논리적으로 묶는 이름 공간입니다.

예를 들어 다음처럼 구성할 수 있습니다.

```text
raw_data_database
├── customers_raw
├── employees_raw
└── orders_raw

process_data_database
├── customers_parquet
└── orders_parquet
```

강의에서는 `raw-data`와 `process-data`라는 두 Glue Database를 만들고, 각각 S3의 해당 폴더를 연결합니다.

콘솔에서 Glue → Data Catalog → Databases로 이동한 뒤, 데이터베이스 이름과 설명을 지정하고 필요하면 기본 S3 위치를 설정합니다.

여기서 데이터베이스는 물리 저장소를 새로 만드는 것이 아니라, 카탈로그의 테이블을 논리적으로 묶고 관리하기 쉽게 하는 구조입니다.

## 3.5 챕터 19. AWS Glue Tables 55:47–1:00:25

Glue Table도 원본 데이터를 복사해 넣는 객체가 아닙니다. 원본 데이터의 위치와 스키마를 표현하는 카탈로그 객체입니다.

예를 들어 S3의 `raw-data/customers/` 안에 CSV가 있으면, Glue Catalog에 `customers` 테이블을 등록해 다음 정보를 저장할 수 있습니다.

- 파일 위치
- CSV 형식과 구분자
- 컬럼 이름
- 컬럼 타입
- 파티션 키
- 분류 정보

원본 CSV는 계속 S3에 있고, Glue Table은 그 파일을 SQL·ETL 엔진이 테이블처럼 이해하도록 돕습니다.

강의에서는 콘솔에서 수동으로 `customers` 테이블을 만드는 과정을 보여줍니다.

1. Glue의 Tables로 이동합니다.
2. **Add table**을 선택합니다.
3. 테이블 이름을 `customers`로 지정합니다.
4. `raw-data` 데이터베이스를 선택합니다.
5. 데이터 저장소로 S3를 선택합니다.
6. 고객 CSV가 있는 경로를 지정합니다.
7. 파일 형식을 CSV, 구분자를 comma로 지정합니다.
8. `customer_id`, `first_name`, `last_name`, `full_name` 같은 컬럼과 타입을 수동으로 정의합니다.
9. 파티션 키는 이 실습에서는 사용하지 않습니다.

이 작업이 끝나면 Catalog에는 CSV 위치와 스키마 정보가 등록되고, 이후 Glue Job이나 Athena에서 이 데이터를 테이블처럼 읽을 수 있습니다.

## 3.6 챕터 20. AWS Glue Crawlers 1:00:26–1:06:09

Glue Crawler는 데이터 저장소를 훑어보고, 파일 형식과 스키마를 추론한 뒤 Glue Data Catalog에 테이블 메타데이터를 만드는 자동화 도구입니다.

Crawler가 하는 일은 다음과 같습니다.

1. S3, JDBC 데이터베이스 같은 데이터 저장소에 연결합니다.
2. 데이터 형식을 판별합니다.
3. 분류기 classifier를 통해 스키마를 추론합니다.
4. Glue Data Catalog에 테이블 정의를 만듭니다.

즉, 사람이 테이블과 컬럼을 하나씩 수동 입력하는 대신, 데이터 위치를 지정하고 Crawler에게 추론을 맡길 수 있습니다.

하지만 자동 추론 결과가 항상 의도와 같다는 보장은 없습니다. 파일의 헤더, 데이터 샘플, 디렉터리 구조, 타입 일관성에 따라 결과가 달라질 수 있습니다. 강의에서는 CSV 헤더를 포함한 별도 파일을 업로드한 이유가 여기에 있습니다.

실습에서는 `orders_headers`를 대상으로 Crawler를 만듭니다.

1. Glue → Crawlers → Create crawler로 이동합니다.
2. 이름을 지정합니다.
3. 아직 Glue Table에 매핑되지 않은 데이터 원천을 선택합니다.
4. S3에서 `raw-data/orders_headers` 경로를 지정합니다.
5. IAM 역할을 선택합니다.
6. 결과 테이블을 저장할 목표 Glue Database로 `raw-data`를 선택합니다.
7. 필요하면 결과 테이블 이름에 접두사를 붙입니다.
8. 온디맨드 실행으로 생성하고 실행합니다.

Crawler가 완료되면 헤더의 컬럼 이름과 데이터 타입을 추론한 Glue Table이 만들어집니다. `employees_headers`에 대해서도 같은 과정을 반복합니다.

## 3.7 챕터 21. Partitions in AWS 1:06:10–1:07:45

파티션 Partition은 큰 데이터를 논리적 기준으로 나누어 저장하고 읽을 때 필요한 부분만 조회하도록 하는 방법입니다.

S3 데이터 레이크에서는 흔히 폴더 경로를 파티션 표현으로 사용합니다.

```text
s3://bucket/events/
  year=2019/
    month=01/
      day=02/
        data.parquet
```

쿼리가 `year=2019 AND month=01 AND day=02`만 요구한다면, 쿼리 엔진은 관련 파티션 경로만 읽고 나머지 데이터는 제외할 수 있습니다. 이를 통해 읽는 데이터량과 쿼리 비용·시간을 줄일 수 있습니다.

강의에서는 파티션을 주로 S3 객체 저장 구조로 설명하지만, 데이터베이스의 대형 테이블에도 파티션 개념을 적용할 수 있습니다.

## 3.8 챕터 22. AWS Glue Connections 1:07:46–1:10:14

Glue Connection은 특정 데이터 저장소에 연결하기 위해 필요한 속성을 담는 Catalog 객체입니다.

예를 들어 Glue Crawler나 ETL Job이 MySQL·Aurora·PostgreSQL에 연결해야 한다면, 다음 정보를 연결 객체에 둘 수 있습니다.

- JDBC URL
- 사용자 이름
- 비밀번호 또는 Secrets Manager 참조
- 네트워크 연결 정보
- 데이터베이스 드라이버 관련 설정

이렇게 하면 ETL 스크립트에 비밀번호와 연결 문자열을 직접 넣지 않아도 됩니다. 또한 개발·테스트·운영 환경별 연결을 분리하고, 자격 증명 변경도 한 곳에서 관리하기 쉬워집니다.

Glue 콘솔에서는 Aurora, JDBC, SaaS, Salesforce, Oracle, Teradata, Snowflake 등 다양한 연결·커넥터를 선택하거나, 필요하면 사용자 정의 커넥터를 만들 수 있습니다.

## 3.9 챕터 23. AWS Glue ETL 1:10:15–1:11:10

Glue ETL은 여러 원천에서 데이터를 추출하고 Extract, 업무 요구에 맞게 변환하고 Transform, 원하는 대상에 적재하는 Load 기능입니다.

Glue Job은 시각적 화면에서 만들 수도 있고 코드로 만들 수도 있습니다. 강의는 주로 Apache Spark 기반의 PySpark ETL 엔진을 중심으로 설명합니다.

기본 흐름은 다음과 같습니다.

```text
Source → Transform → Target
```

예를 들어 Glue Data Catalog의 고객 테이블을 읽어서 현재 적재 시간을 추가하고, Parquet 형식으로 S3의 처리 영역에 저장할 수 있습니다.

## 3.10 챕터 24. AWS Glue DPUs 1:11:11–1:11:51

DPU는 Data Processing Unit의 약자입니다. Glue 작업의 처리 자원 크기를 표현하는 단위입니다.

강의에서 설명하는 기준으로는 1 DPU가 4 vCPU와 16GB 메모리에 해당합니다. 따라서 32GB 메모리가 필요한 계산 문제라면 단순 기준으로 2 DPU를 떠올릴 수 있습니다.

[번역 주석] Glue의 실제 워커 유형·자원 규격은 버전과 워커 타입에 따라 달라질 수 있습니다. 이 문장은 원문 강의의 학습용 단순화 설명입니다.

## 3.11 챕터 25. AWS Glue Bookmarks 및 Visual ETL 실습 1:11:52–1:19:43

Glue Job Bookmark는 이미 처리한 데이터를 추적하는 기능입니다. 대량 배치 작업에서 기존 데이터를 다시 전부 처리하지 않고, 새로 들어온 데이터만 증분 처리할 때 사용합니다.

예를 들어 시간별·일별 폴더에 데이터가 쌓인다면 Bookmark를 이용해 이전 실행에서 처리한 파일·범위를 기억하고 새 데이터만 처리하도록 만들 수 있습니다. 그래서 데이터 도착 경로와 파티션 구조가 중요해집니다.

실습에서는 Glue Studio Visual ETL로 다음 작업을 만듭니다.

1. Source로 Glue Catalog의 `raw-data` 데이터베이스와 `customers` 테이블을 선택합니다.
2. Transform으로 현재 시각을 `load_time` 컬럼에 넣습니다.
3. Target으로 S3를 선택합니다.
4. 결과 형식은 Parquet, 압축은 Snappy로 지정합니다.
5. 결과는 `process-data` 경로에 저장합니다.
6. 결과 테이블을 `process-data` Glue Database에 생성합니다.
7. `load_time`을 파티션 키로 지정합니다.

Visual ETL은 화면에서 노드를 연결하지만, 내부적으로는 PySpark 코드가 생성됩니다. 강의에서는 생성된 스크립트를 확인하고, IAM 역할·DPU 수·스크립트 저장 위치·임시 경로·로그 경로를 설정한 뒤 Job을 실행합니다.

성공하면 다음 결과를 확인할 수 있습니다.

- Glue Catalog의 `process-data` 데이터베이스에 새 테이블 생성
- S3의 `process-data` 위치에 Parquet 파일 생성
- `load_time=...` 형식의 파티션 경로 생성

강사는 Glue Studio에서 다른 원천을 추가하고, 조인 같은 변환도 시도해 보라고 제안합니다.

## 3.12 챕터 26. AWS Glue Scheduling 1:19:44–1:26:45

Glue에서 가장 중요한 실행 제어 개념은 Trigger입니다.

Trigger는 다음 기준으로 Job이나 Crawler를 시작할 수 있습니다.

- 시간 기반 스케줄: 매시간, 매일, 매주 등
- 이벤트 기반: 다른 Job·Crawler가 성공했을 때
- 온디맨드: 사람이 직접 실행할 때

Glue Workflow는 Crawler, Job, Trigger를 묶어 시각적으로 복잡한 작업 흐름을 만들 수 있습니다.

예를 들면 다음과 같습니다.

```text
Orders Crawler 성공
  → Employees Crawler 실행
    → ETL Job 실행
```

단순히 Glue Job 하나를 정해진 시간에 실행할 때는 Glue Job의 Schedule 설정만으로 충분할 수 있습니다. 하지만 Glue 이외에 EMR, Lambda, Kinesis 등 여러 서비스를 함께 조정해야 한다면 AWS Step Functions, Amazon EventBridge, Amazon MWAA Managed Workflows for Apache Airflow 같은 별도 오케스트레이션 도구가 더 적합할 수 있습니다.

강의 실습에서는 매시간 5분에 실행하는 스케줄을 만들고, Crawler 성공 이후 다음 Crawler나 ETL Job을 실행하는 Trigger를 만듭니다. 또한 온디맨드 Trigger와 ETL Job을 연결한 간단한 Glue Workflow도 만듭니다.

## 3.13 챕터 27. AWS Glue Data Quality 1:26:46–1:28:12

AWS Glue Data Quality는 데이터 품질을 규칙으로 검사하는 기능입니다. 강의에서는 DQDL Data Quality Definition Language과 오픈소스 Deequ를 언급합니다.

예를 들어 다음 같은 규칙을 정의할 수 있습니다.

- 필수 컬럼이 존재해야 함
- 특정 컬럼은 null이 아니어야 함
- 레코드 수가 최소 기준 이상이어야 함
- 값이 허용된 범위 안에 있어야 함
- 기본 키 후보의 중복이 없어야 함

Glue 콘솔에서 테이블을 선택하고 추천 규칙을 만들 수 있습니다. 일 단위로 데이터가 들어올 때 필수 컬럼이 사라지거나 규칙이 깨지면 경고를 내고, 문제를 수정한 뒤 파이프라인의 다음 단계로 진행하게 만들 수 있습니다.

## 3.14 챕터 28. AWS Glue DataBrew 1:28:13–1:29:51

AWS Glue DataBrew는 코드를 많이 쓰지 않고도 데이터를 정리·정규화·변환할 수 있는 시각적 데이터 준비 도구입니다.

강사는 이를 비유적으로 “Glue 안의 스프레드시트 같은 도구”로 설명합니다. 데이터 분석가나 데이터 과학자가 CSV 같은 데이터를 보고 다음 작업을 하기 좋습니다.

- 컬럼 정리
- 결측치 처리
- 값 변환
- 형식 정규화
- 간단한 데이터 준비

강의에서는 S3의 CSV를 불러오고, 헤더가 있다는 설정을 선택한 뒤 시각적 규칙을 적용하는 흐름을 보여줍니다.

> **[2026 최신화]** DataBrew는 현재도 공식 문서상 서버리스 시각적 데이터 준비 도구로 제공됨. 다만 시험에서는 “코드 없이 시각적으로 데이터 준비”라는 역할 정도로 잡고, 핵심 시간은 Glue ETL, Catalog, Athena, Redshift, 보안·거버넌스에 우선 배분하는 편이 맞음. [공식 근거 5][5]

## 3.15 챕터 29. Glue Clean Up Work 1:29:52–1:30:56

Glue 실습이 끝난 뒤에는 데모를 위해 넣었던 헤더 포함 데이터셋을 S3에서 삭제합니다. 이후 강의에서 원본 데이터 구조와 혼동하지 않기 위해서입니다.

또한 더 이상 쓰지 않는 IAM 역할과 Glue 실습용 CloudFormation 스택을 삭제해 불필요한 권한과 리소스를 남기지 않도록 합니다.

# 4. Amazon Athena

## 4.1 챕터 30. Amazon Athena 1:30:57–1:35:10

Amazon Athena는 표준 SQL을 사용해 S3의 데이터를 대화형으로 분석할 수 있는 서버리스 쿼리 서비스입니다.

핵심은 다음과 같습니다.

- S3에 있는 데이터를 SQL로 조회
- 서버·클러스터를 직접 관리하지 않음
- 쿼리 실행량 기반 비용 모델
- AWS Glue Data Catalog와 통합
- 필요에 따라 다른 데이터 원천과의 연동 및 페더레이티드 쿼리 가능

Athena를 사용하면 사용자는 SQL을 제출하고 결과를 받습니다. 데이터 위치·스키마는 Glue Catalog에서 찾고, 실제 컴퓨팅 인프라는 AWS가 관리합니다.

강의에서는 Athena 엔진 버전에 따라 Presto와 Trino를 언급합니다. 사용자는 SQL 구문과 서비스 특성에 집중하고, 내부 클러스터 운영은 AWS에 맡기는 구조입니다.

[번역 주석] 엔진의 구체적인 명칭·버전·지원 기능은 변할 수 있으므로, 현재 환경에서 엔진 정보를 확인해야 합니다.

> **[2026 최신화]** Athena Engine version 3는 Trino와 Presto 오픈소스 프로젝트의 개선을 반영하는 엔진임. 강의의 “v2는 Presto, v3는 Trino”라는 암기 문장은 내부 계보를 과도하게 단순화한 것임. 시험에서는 **Athena는 S3와 Data Catalog를 기반으로 SQL을 실행하는 서버리스 분석 서비스**라는 역할, 엔진 v3 호환성 차이, Iceberg 테이블 지원을 이해하는 쪽이 더 중요함. [공식 근거 3][3]

Athena의 전형적 처리 흐름은 다음과 같습니다.

```text

  → 서버리스 엔진이 쿼리를 처리
  → 결과를 콘솔과 지정 S3 결과 경로에 반환
```

# 5. Amazon Redshift

## 5.1 챕터 33. Amazon Redshift 1:54:48–2:15:44

Amazon Redshift는 대규모 데이터를 분석하는 완전관리형 데이터 웨어하우스 서비스입니다. SQL로 분석 쿼리를 실행하며, BI·리포팅·대규모 집계에 적합합니다.

[번역 주석] 원문 자동 자막에 `OLTP`로 나온 대목은 문맥상 `OLAP Online Analytical Processing`을 의미합니다. Redshift는 분석용 OLAP 워크로드에 쓰는 서비스입니다.

대표적인 사용 사례는 다음과 같습니다.

- 대규모 분석 워크로드
- BI와 대시보드
- 여러 데이터 웨어하우스·분석 데이터를 통합
- 주식·시장·행동·소셜 트렌드 분석
- 데이터 레이크와 결합한 분석

### Redshift 아키텍처

클러스터형 Redshift는 일반적으로 다음 구성으로 설명할 수 있습니다.

- **Leader node**: 클라이언트 연결을 받고, 쿼리 계획·조정을 담당
- **Compute node**: 실제 데이터 저장·처리·병렬 계산을 수행
- **Slice**: 컴퓨트 노드 안에서 병렬 계산을 나누는 단위
- **JDBC/ODBC client**: 클러스터에 SQL을 보내는 클라이언트

RA3 노드 계열에서는 관리형 스토리지와 컴퓨팅을 분리해 확장할 수 있다는 점도 언급합니다. 저장 용량은 늘어나지만 컴퓨팅 요구는 크지 않은 상황에서 이 분리가 유용할 수 있습니다.

### 내구성과 재해 복구

Redshift는 클러스터 내부 복제와 스냅샷을 통해 내구성을 지원합니다. 자동 스냅샷을 S3에 보관하고, 장애나 손상 시 복구에 활용할 수 있습니다.

다른 리전으로 스냅샷을 복사해 재해 복구 구성을 만들 수도 있습니다. 강의에서는 대상 리전에 KMS 키와 Snapshot Copy Grant를 준비하고, 원본 리전에서 스냅샷 복사를 활성화하는 흐름을 설명합니다.

### 확장 방식

강의는 세 가지 확장 접근을 소개합니다.

- **Elastic resize**: 같은 인스턴스 타입 안에서 노드를 빠르게 늘리거나 줄이는 방식
- **Classic resize**: 인스턴스 타입과 노드 수를 바꾸는 더 큰 변경 방식
- **Snapshot and restore**: 새 클러스터를 복원해 트래픽을 전환한 뒤 기존 클러스터를 조정하는 방식

강의는 클래식 리사이즈가 더 오래 걸리고, 가용성·읽기 전용 상태 같은 운영 영향을 고려해야 한다고 설명합니다.

### 배포 키 Distribution Style

Redshift에서는 데이터를 컴퓨트 노드와 슬라이스에 어떻게 분산할지 선택할 수 있습니다.

#### AUTO

Redshift가 데이터와 쿼리 특성을 보고 분산 방식을 선택하게 하는 옵션입니다. 확실한 근거가 없다면 자동 선택을 먼저 고려할 수 있습니다.

#### EVEN

행을 라운드로빈 방식으로 균등하게 나눕니다. 특정 조인 키에 맞춘 배치가 필요하지 않고, 데이터 특성도 잘 모를 때 사용할 수 있습니다.

#### KEY

같은 분산 키 값을 가진 행이 같은 위치에 배치되도록 합니다. 큰 테이블끼리 특정 조인 키로 자주 조인할 때, 데이터 재분산을 줄이는 데 유용할 수 있습니다.

#### ALL

테이블 전체를 각 컴퓨트 노드에 복제합니다. 국가 코드 같은 작은 룩업·차원 테이블을 여러 대형 테이블과 자주 조인할 때 유리할 수 있습니다. 큰 테이블에 적용하면 저장과 유지 비용이 커질 수 있으므로 주의해야 합니다.

### COPY와 UNLOAD

`COPY`는 Redshift로 데이터를 적재하는 대표 명령입니다. S3, DynamoDB, EMR, 원격 호스트 등의 데이터를 Redshift 테이블로 읽어 올 수 있습니다.

강의는 `COPY`가 압축·암호화된 파일을 처리하고, 병렬로 데이터를 적재할 수 있다고 설명합니다. S3에 새 데이터가 도착할 때 자동 적재 기능을 쓰는 패턴도 언급합니다.

`UNLOAD`는 쿼리 결과를 S3로 내보내는 명령입니다. 텍스트, JSON, Parquet 등 형식으로 저장할 수 있고, S3 서버 측 암호화를 사용할 수 있습니다.

### Zero-ETL, Streaming, Integration

강의는 Aurora·RDS와 Redshift 간 Zero-ETL 통합을 언급합니다. 이는 운영 데이터 변경을 Redshift 분석 환경에 지속적으로 반영하려는 경우의 선택지입니다.

Kinesis 또는 MSK에서 Redshift로 스트리밍 데이터를 수집하는 방식도 언급합니다.

### VACUUM

`VACUUM`은 삭제된 행으로 인해 생긴 공간을 회수하고, 필요에 따라 정렬 상태를 정리하는 명령입니다.

강의에서 소개하는 유형은 다음과 같습니다.

- `VACUUM FULL`: 삭제 공간 회수와 정렬을 함께 수행
- `VACUUM DELETE ONLY`: 삭제된 행으로 생긴 공간만 회수
- `VACUUM SORT ONLY`: 정렬만 수행
- `VACUUM REINDEX`: 인터리브드 정렬 키 관련 재인덱싱

### Workload Management WLM

WLM은 긴 쿼리와 짧은 쿼리가 같은 자원을 두고 서로 막지 않도록, 쿼리를 큐와 슬롯으로 관리하는 기능입니다.

강의의 메시지는 다음과 같습니다.

- 자동 WLM은 쿼리별 자원 요구와 동시성을 AWS가 조정하므로 일반적으로 우선 고려
- 수동 WLM은 특별한 이유가 있을 때만 세밀하게 구성
- 쿼리 우선순위, 사용자 그룹, 쿼리 그룹, 모니터링 규칙 등을 활용 가능
- 짧은 쿼리를 빠르게 처리하기 위해 Short Query Acceleration SQA를 활용할 수 있음
- 동시 사용자가 급증하면 Concurrency Scaling으로 추가 용량을 사용할 수 있음

### RA3, Spectrum, Data Sharing, Federated Query, Serverless

- **RA3 nodes**: 컴퓨팅과 관리형 스토리지를 독립적으로 확장하는 노드 계열
- **Redshift Spectrum**: Redshift에서 S3 데이터 레이크의 외부 테이블을 조회하는 기능
- **Data sharing**: 데이터를 복제하지 않고 다른 Redshift 클러스터·계정·리전에 실시간 접근을 제공하는 방식
- **Federated query**: Redshift에서 RDS·Aurora 등의 운영 데이터에 쿼리하는 방식
- **Redshift Serverless**: 클러스터를 직접 프로비저닝하지 않고 워크로드에 따라 자동 확장하는 형태

강의는 시험 대비 우선순위로 `COPY`, 분산 스타일, WLM, VACUUM, Redshift Spectrum, 서버리스와 클러스터형 선택을 강조합니다.

# 6. Amazon EMR

## 6.1 챕터 36. Amazon EMR 2:38:41–2:44:06

Amazon EMR은 Hadoop, Spark, Hive, Trino, Flink 같은 빅데이터 프레임워크를 실행하기 쉽게 만든 관리형 클러스터 플랫폼입니다.

EMR은 다른 AWS 서비스와 통합하며, 특히 Glue Data Catalog를 메타스토어로 활용할 수 있습니다.

전통적인 EMR 클러스터 구성은 다음과 같습니다.

- **Primary node 또는 Master node**: 클러스터를 조정하고 다른 노드의 작업을 관리
- **Core node**: 작업을 실행하고 HDFS에 데이터를 저장
- **Task node**: 작업만 실행하고 HDFS 저장은 하지 않는 선택적 노드

강의는 EMR의 자원 관리 계층으로 YARN을 언급하고, 저장은 HDFS 또는 S3와 결합할 수 있다고 설명합니다. 오늘날 흔한 패턴은 데이터를 S3에 두고, EMR은 필요할 때 S3 데이터를 읽어 대규모 처리를 수행하는 방식입니다.

### EMR 실행 형태

- **EMR on EC2**: EC2 기반 클러스터를 구성해 프레임워크 실행
- **EMR Serverless**: 서버를 직접 프로비저닝하지 않고 Spark·Hive·Trino 등의 작업 실행
- **EMR on EKS**: Kubernetes 기반 EKS에서 EMR 런타임을 실행

강의는 EMR Serverless도 애플리케이션을 실제로 종료·삭제하지 않으면 비용이 남을 수 있다는 점을 언급합니다. 또한 EMR on EKS는 컨테이너·Kubernetes 환경 위에서 빅데이터 프레임워크를 실행하는 선택지라고 설명합니다.

> **[원문 정정 및 2026 최신화]** “EMR Serverless 애플리케이션을 종료하지 않으면 항상 비용이 든다”는 식의 설명은 부정확함. 기본 Auto-stop은 유휴 애플리케이션을 자동 정지할 수 있음. 다만 `initialCapacity`로 미리 준비된 worker를 유지하면 애플리케이션이 `STARTED` 상태인 동안 유휴여도 비용이 발생할 수 있음. 즉 **일반 실행 자원과 warm capacity 비용을 구분**해야 함. [공식 근거 4][4]

EMR의 대표 사용처는 다음과 같습니다.

- 대규모 배치 수집·변환
- Spark 기반 대규모 분석
- Trino·Presto 기반 SQL 분석
- Spark Structured Streaming이나 Flink 기반 스트리밍 처리

AWS Step Functions는 EMR 작업 시작·완료·분기·재시도 같은 흐름을 조정하는 오케스트레이션 도구로 활용할 수 있습니다.

# 7. Amazon Kinesis

## 7.1 챕터 38. Amazon Kinesis 3:13:55–3:20:44

Amazon Kinesis는 실시간 스트리밍 데이터를 수집·버퍼링·처리하는 AWS 서비스군입니다. 많은 원천에서 들어오는 데이터를 낮은 지연 시간으로 처리하는 용도로 사용합니다.

강의에서는 Kinesis의 세 가지 핵심 구성을 소개합니다.

- Amazon Kinesis Data Streams
- Amazon Data Firehose
- Amazon Managed Service for Apache Flink

### Kinesis Data Streams

Kinesis Data Streams는 여러 개의 **Shard**로 구성된 스트림입니다.

- **Stream**: 여러 Shard의 집합
- **Shard**: 순서가 있는 데이터 레코드 스트림의 단위
- **Record**: 스트림에 저장되는 하나의 데이터 단위
- **Data blob**: 레코드 안의 실제 데이터
- **Partition key**: 어떤 Shard에 기록할지를 결정하는 키
- **Sequence number**: 특정 Shard 안에서 레코드의 순서를 나타내는 번호
- **Producer**: 스트림에 레코드를 쓰는 애플리케이션
- **Consumer**: 스트림에서 레코드를 읽는 애플리케이션

단순화하면 다음과 같습니다.

```text
Producer → Kinesis Data Stream의 Shard들 → Consumer → S3 / Lambda / Redshift / 기타 대상
```

Kinesis Client Library KCL은 분산 소비 작업의 체크포인트와 샤드 처리 같은 복잡한 부분을 대신 관리해 줍니다. 애플리케이션이 Kinesis에서 안정적으로 데이터를 읽고, 처리 위치를 관리해야 한다면 KCL을 후보로 생각합니다.

SDK나 CLI를 이용해 직접 `PutRecord`, `PutRecords` 호출로 데이터를 쓸 수도 있습니다. 하지만 생산·소비 처리의 복잡도를 줄이려면 라이브러리와 관리형 통합을 활용하는 것이 일반적입니다.

### Enhanced Fan-Out

여러 Consumer가 같은 스트림을 읽을 때, 표준 소비 방식에서는 Shard 읽기 처리량을 Consumer들이 공유합니다. Consumer 수가 늘면 지연이 커질 수 있습니다.

Enhanced Fan-Out은 Consumer별로 전용 읽기 처리량을 제공해 이러한 경쟁을 줄입니다. 대신 추가 비용을 고려해야 합니다.

시험에서는 다음 식으로 판단합니다.

- 여러 Consumer가 한 스트림을 동시에 읽음
- 각 Consumer에 낮은 지연과 독립적 처리량이 필요함
- Consumer 간 처리량 경합을 줄여야 함

이런 조건이면 Enhanced Fan-Out을 검토합니다.

### Amazon Data Firehose

Data Firehose는 스트리밍 데이터를 S3, Redshift, OpenSearch 같은 대상 저장소로 적재하는 관리형 전달 서비스입니다.

Kinesis Data Streams를 쓸 때는 일반적으로 Consumer 애플리케이션을 직접 구현해야 합니다. 반면 Firehose는 전달·버퍼링·적재를 관리형으로 처리하므로, 스트리밍 데이터를 S3에 적재해야 하는 단순한 요구에서 특히 유용합니다.

시험에서 `스트리밍 데이터를 별도 소비자 구현 부담 없이 S3로 전달` 같은 표현이 나오면 Firehose를 우선 후보로 봅니다.

### Amazon Managed Service for Apache Flink

Amazon Managed Service for Apache Flink는 Apache Flink로 실시간 스트리밍 처리를 할 때 기반 인프라를 관리형으로 제공하는 서비스입니다.

실시간 스트림에서 상태를 가진 변환, 윈도우 집계, 이벤트 시간 처리 같은 Flink 처리 모델이 필요하다면 이 서비스를 고려할 수 있습니다. EMR에서도 Flink를 운영할 수 있지만, 이 서비스는 Flink 실행 인프라 관리를 더 줄이는 방향입니다.

# 8. Amazon MSK

## 8.1 챕터 40. Amazon MSK 3:35:xx–3:37:xx

Amazon MSK Managed Streaming for Apache Kafka는 Apache Kafka를 AWS에서 관리형으로 운영하는 서비스입니다.

Kafka를 직접 운영하면 브로커, Zookeeper 또는 KRaft 구성, 패치, 확장, 모니터링, 장애 조치 같은 운영 부담이 큽니다. MSK는 이 중 인프라 운영 부담을 줄이고 Kafka API·생태계를 활용하려는 서비스입니다.

핵심 개념은 다음과 같습니다.

- **Broker**: Kafka 데이터를 저장·전달하는 서버
- **Topic**: 메시지를 논리적으로 구분하는 채널
- **Partition**: Topic을 병렬 처리할 수 있게 나눈 단위
- **Producer**: Topic에 메시지를 쓰는 애플리케이션
- **Consumer group**: Topic을 나누어 읽는 소비자 집합
- **Offset**: 파티션에서 메시지를 어디까지 읽었는지 나타내는 위치

## 8.2 챕터 41. Amazon MSK vs Kinesis 3:37:xx–3:39:xx

Kinesis와 MSK는 모두 스트리밍 데이터 처리와 관련되지만, 선택 이유가 다릅니다.

### Kinesis를 먼저 고려할 상황

- AWS 네이티브 관리형 스트리밍을 원함
- 인프라 운영 부담을 최소화하려 함

### MSK를 먼저 고려할 상황

- 기존에 Apache Kafka를 사용하고 있음
- Kafka API, 클라이언트, 커넥터, 운영 방식과 호환되어야 함
- Kafka Topic·Partition·Consumer Group 모델을 유지해야 함
- AWS 안에서 Kafka를 운영하되 관리형 서비스를 사용하고 싶음

시험에서는 “Kafka 호환성”, “기존 Kafka 워크로드 마이그레이션”, “Kafka 클라이언트와 생태계” 같은 단서가 있으면 MSK를 검토하고, AWS 관리형 스트림과 목적지 적재 중심이라면 Kinesis를 먼저 검토합니다.

# 9. Amazon OpenSearch Service

## 9.1 챕터 42. Amazon OpenSearch 3:39:xx–3:43:xx

Amazon OpenSearch Service는 검색, 로그 분석, 관측성, 텍스트 기반 탐색 등에 사용하는 관리형 검색·분석 서비스입니다.

OpenSearch의 핵심은 관계형 테이블을 조인해서 읽는 방식보다, 문서를 색인 Index하고 빠르게 검색하는 방식에 있습니다.

주요 용어는 다음과 같습니다.

- **Index**: 비슷한 문서를 저장하는 논리적 컬렉션
- **Document**: JSON 형식의 하나의 데이터 레코드
- **Field**: 문서 안의 속성
- **Shard**: 인덱스를 나누어 저장·처리하는 단위
- **Primary shard**: 원본 샤드
- **Replica shard**: 복제본 샤드
- **Cluster**: 여러 노드가 모인 검색 시스템
- **Node**: 클러스터 구성 단위
- **Mapping**: 필드 타입과 색인 방식을 정의하는 스키마 성격의 설정

OpenSearch는 전문 검색, 로그 분석, 보안 이벤트 탐색, 애플리케이션 검색, 대시보드에 적합합니다.

강의는 OpenSearch가 데이터를 수집·저장·색인·검색·시각화하는 흐름을 강조합니다.

예를 들어 제목에 `Mars`가 들어간 영화를 검색하면, `movies` 인덱스의 문서 중 제목 필드가 조건에 맞는 결과를 JSON으로 받습니다.

Dashboards의 Discover 화면에서는 인덱스 패턴을 만들고, 키워드를 검색해 관련 문서를 확인합니다. 이 실습은 OpenSearch가 문서를 색인해 빠르게 찾는 구조를 보여주기 위한 것입니다.

OpenSearch 도메인은 비용이 발생할 수 있으므로, 학습이 끝나면 반드시 삭제합니다.

# 11. Amazon DynamoDB

## 11.1 챕터 45. Amazon DynamoDB 3:50:xx–3:52:xx

Amazon DynamoDB는 완전관리형 NoSQL 데이터베이스입니다. 낮은 지연 시간으로 키 기반 데이터를 읽고 쓰는 데 적합합니다.

관계형 데이터베이스처럼 조인과 정규화를 중심으로 설계하기보다, 애플리케이션이 실제로 어떤 조회를 해야 하는지 Access Pattern을 먼저 정의하고 키 구조를 설계하는 방식이 중요합니다.

기본 용어는 다음과 같습니다.

- **Table**: 아이템을 담는 논리 저장소
- **Item**: 하나의 레코드에 해당하는 데이터 단위
- **Attribute**: 아이템의 속성, 관계형 DB의 컬럼과 비슷한 개념
- **Primary key**: 아이템을 식별하는 키
- **Partition key**: 데이터 분산과 조회의 기준이 되는 키
- **Sort key**: 같은 Partition key 안에서 항목을 구분·정렬하는 키

Primary key는 다음 두 가지 형태 중 하나입니다.

```text
단순 기본 키: Partition key만 사용
복합 기본 키: Partition key + Sort key 사용
```

단순 기본 키를 쓰면 같은 Partition key 값으로 새 아이템을 만들 수 없습니다. 복합 기본 키를 쓰면 Partition key가 같아도 Sort key가 다르면 여러 아이템을 저장할 수 있습니다.

예를 들어 고객의 이력 데이터를 시간별로 저장하려면 다음처럼 설계할 수 있습니다.

```text
PK = customer_id
SK = timestamp
```

이러면 고객 한 명에 대해 시간별 여러 레코드를 보관하고, 특정 기간이나 최신 기록을 조회할 수 있습니다.

## 11.2 챕터 46. DynamoDB Indexes 3:52:xx–3:54:29

DynamoDB의 Secondary Index는 기본 키 이외의 조회 패턴을 지원하기 위해 만듭니다.

### LSI Local Secondary Index

LSI는 Base Table과 **같은 Partition key**를 사용하고, Sort key만 다르게 정의합니다.

예를 들어 기본 키가 `customer_id + timestamp`일 때, 같은 고객 안에서 `country` 기준으로 조회하고 싶다면 LSI를 고려할 수 있습니다.

핵심 제약은 Partition key를 바꿀 수 없다는 점입니다.

### GSI Global Secondary Index

GSI는 Base Table과 다른 Partition key를 사용할 수 있습니다. 따라서 원래 테이블에서 `customer_id`로 저장했더라도, `country`를 Partition key로 하는 조회가 자주 필요하면 GSI를 만들 수 있습니다.

예를 들어 다음 요구가 있다고 가정합니다.

```text
기본 조회: 고객 ID로 특정 고객의 이력 조회
추가 조회: 특정 국가의 모든 고객 조회
```

이 경우 `country`를 Partition key로 하는 GSI를 고려합니다.

강의의 핵심 판단 기준은 매우 단순합니다.

- 다른 Partition key로 조회해야 함: **GSI**
- 기본 테이블과 동일한 Partition key 안에서 다른 Sort key 조회가 필요함: **LSI**

강의는 GSI가 별도 인덱스 저장 구조를 갖고, 처리량을 기본 테이블과 독립적으로 다룰 수 있다는 점도 언급합니다. 반면 LSI는 기본 테이블의 Partition key 제약을 공유합니다.

## 11.3 챕터 47. DynamoDB Capacity Units 3:54:30–3:57:00

DynamoDB의 처리량은 읽기와 쓰기 용량 단위로 표현합니다.

- **RCU Read Capacity Unit**: 읽기 처리량 단위
- **WCU Write Capacity Unit**: 쓰기 처리량 단위

강의의 단순 계산 기준은 다음과 같습니다.

### RCU

크기가 4KB 이하인 아이템을 기준으로,

- 강한 일관성 읽기 Strongly Consistent Read 초당 1회 = 1 RCU
- 최종 일관성 읽기 Eventually Consistent Read 초당 2회 = 1 RCU

아이템이 4KB보다 크면, 4KB 단위로 필요한 RCU가 증가합니다.

예를 들어 6KB 아이템을 강한 일관성으로 초당 1회 읽으면 4KB를 넘으므로 2 RCU가 필요하다는 식으로 계산합니다.

### WCU

크기가 1KB 이하인 아이템을 초당 1회 쓰는 데 1 WCU가 필요합니다.

아이템이 1KB보다 크면 1KB 단위로 반올림해 WCU가 증가합니다. 예를 들어 3KB 아이템을 초당 1회 쓰려면 3 WCU가 필요하다는 식입니다.

시험에서는 데이터 크기, 읽기 일관성, 초당 읽기·쓰기 횟수를 주고 RCU·WCU를 계산하게 할 수 있으므로, 계산 기준을 알아야 합니다.

[번역 주석] 세부 단위와 비용 구조는 실제 시험 가이드·공식 문서로 재확인해야 합니다. 강의의 설명은 기본 계산 패턴을 익히기 위한 것입니다.

## 11.4 챕터 48. DynamoDB Modes 3:57:01–3:57:48

DynamoDB 용량 모드는 크게 두 가지입니다.

### Provisioned Mode

사용자가 예상 처리량에 맞춰 RCU와 WCU를 지정합니다. 트래픽 패턴이 비교적 예측 가능하면 비용을 통제하기 쉽습니다. Auto Scaling을 함께 써서 일정 범위 안에서 용량을 조절할 수 있습니다.

### On-demand Mode

사용자가 RCU·WCU를 미리 계산하지 않아도 되고, DynamoDB가 워크로드 변화에 맞춰 처리량을 조정합니다. 예측하기 어려운 트래픽이나 급격한 변동에 편하지만, 일반적으로 비용 최적화 관점에서는 Provisioned Mode와 비교가 필요합니다.

시험에서는 다음처럼 판단합니다.

- 예측 가능한 안정적 트래픽, 비용 최적화: Provisioned + Auto Scaling 검토
- 예측이 어려운 급격한 변동, 운영 단순화: On-demand 검토

## 11.5 챕터 49. PartiQL 3:57:49–3:58:20

PartiQL은 DynamoDB에서 SQL과 유사한 문법으로 데이터를 조회·삽입·수정·삭제할 수 있게 하는 쿼리 언어입니다.

전통적인 DynamoDB 접근은 Partition key, Sort key, Query, Scan 같은 API 중심입니다. PartiQL은 SQL에 익숙한 사용자에게 보다 친숙한 방식으로 DynamoDB를 다루는 선택지를 제공합니다.

## 11.8 챕터 52. DynamoDB Data Engineering 3:59:36–4:02:07

강의는 DynamoDB가 데이터 엔지니어링 파이프라인의 운영 계층으로 어떻게 연결될 수 있는지 예시를 듭니다.

### DynamoDB 변경을 분석계로 보내는 패턴

```text
DynamoDB
  → DynamoDB Streams
  → Kinesis Data Streams
  → Redshift 적재
  → QuickSight 대시보드
```

이런 구조에서는 DynamoDB가 운영 애플리케이션의 빠른 읽기·쓰기를 담당하고, 변경 이벤트는 스트림을 통해 분석용 Redshift로 옮겨집니다. Redshift에서는 분석에 맞는 구조로 가공하고 QuickSight가 대시보드를 제공합니다.

### 큰 객체는 S3에, DynamoDB에는 위치만 저장하는 패턴

DynamoDB 아이템에 이미지·대형 파일·큰 BLOB을 직접 넣는 것은 적합하지 않을 수 있습니다. 이런 경우 큰 객체는 S3에 저장하고, DynamoDB에는 S3 객체 키나 URI 같은 포인터만 저장합니다.

```text
클라이언트 요청
  → DynamoDB에서 객체 메타데이터·S3 위치 조회
  → S3에서 실제 큰 객체 조회
```

이 패턴은 DynamoDB의 아이템 크기·비용·처리량 제약을 피하고, S3를 대형 객체 저장소로 활용하는 전형적인 설계입니다.

## 11.9 챕터 53. DynamoDB Hands-on 4:02:08–4:16:38

실습에서는 먼저 Partition key만 있는 `customers` 테이블을 만듭니다. `customer_id`를 기본 키로 쓰고, `full_name`, `country` 같은 속성을 가진 아이템을 넣습니다.

같은 `customer_id`로 새 아이템을 만들려고 하면 기본 키가 중복되므로 생성이 실패합니다. 하지만 기존 아이템을 업데이트하는 것은 가능합니다.

다음으로 Partition key와 Sort key를 함께 사용하는 테이블을 만듭니다.

```text
PK = customer_id
SK = timestamp
```

같은 고객 ID라도 타임스탬프가 다르면 여러 이력 레코드를 저장할 수 있습니다. 콘솔의 Query 기능으로 고객 ID를 기준으로 조회하고, Sort key 조건으로 특정 시점·범위를 필터링하는 과정을 확인합니다.

그 다음 LSI와 GSI를 생성해 봅니다.

- LSI는 같은 고객 ID 안에서 다른 정렬·조회 기준을 제공
- GSI는 국가 같은 다른 Partition key로 조회 가능

실습에서 국가를 GSI Partition key로 두면, 고객 ID를 몰라도 `country = France` 같은 조건으로 해당 국가 고객을 조회할 수 있습니다. 단, Partition key 값은 정확히 일치해야 하며 대소문자·데이터 품질 문제가 결과에 영향을 줄 수 있음을 보여줍니다.

# 13. Amazon S3와 데이터 레이크

## 13.1 챕터 58. Amazon S3 4:20:22–4:20:59

Amazon S3 Simple Storage Service는 객체 스토리지 서비스입니다. 매우 큰 규모의 데이터를 저장하고, 필요할 때 인터넷과 AWS 서비스에서 접근할 수 있습니다.

이 강의에서는 S3를 AWS 데이터 엔지니어링의 중심 저장소로 봅니다.

## 13.2 챕터 59. S3 and Data Engineering 4:21:00–4:21:46

S3는 데이터 엔지니어링에서 다음 용도로 쓰입니다.

- 데이터 수집과 원시 데이터 저장
- 데이터 레이크 구성
- Glue·EMR·Athena 처리의 입출력 저장소
- Redshift·DynamoDB·RDS 등의 백업·아카이브
- 머신러닝 학습 데이터 저장
- S3 이벤트 기반 후속 처리 트리거

S3에 객체가 도착하면 Lambda·EventBridge 등으로 후속 작업을 시작할 수 있습니다. 따라서 S3는 단순 파일 저장소가 아니라 데이터 파이프라인의 중심 접점이 됩니다.

## 13.3 챕터 60. S3 and Data Lakes 4:21:47–4:24:17

S3 기반 데이터 레이크는 정형·반정형·비정형 데이터를 원래 모습에 가깝게 저장하고, 필요한 도구를 올려 분석하는 구조입니다.

강의는 Medallion Architecture를 소개합니다. 이름은 조직마다 다를 수 있지만, 보통 데이터 품질과 가공 수준에 따라 계층을 나눕니다.

```text
Raw 또는 Bronze
  → Decoded / Cleaned 또는 Silver
    → Enriched / Curated 또는 Gold
```

예시 아키텍처는 차량 IoT 데이터를 다룹니다.

```text
차량 IoT 데이터
  → Kinesis
  → Lambda / Firehose 변환
  → S3 Raw 저장
  → Decode 단계 S3 저장
  → EMR 배치 보강·정제
  → Enriched S3 저장
  → Athena 분석
  → SageMaker 데이터 과학
  → DynamoDB는 빠른 조회용 운영 저장소
```

여기서 S3는 데이터 레이크의 각 계층을 저장하고, Kinesis는 스트림 이동을 담당하며, EMR은 대규모 변환을, Athena는 분석 질의를, SageMaker는 데이터 과학 작업을 담당합니다.

## 13.4 챕터 61. S3 Lifecycle Policy 4:24:18–4:25:07

S3 Lifecycle Policy는 객체의 나이·접근 빈도·접두사·태그 등을 기준으로 저장 클래스를 바꾸거나 삭제하는 자동화 규칙입니다.

예를 들어 다음 정책을 만들 수 있습니다.

- 생성 후 30일 동안 접근하지 않은 객체를 더 저렴한 저장 클래스로 전환
- 일정 기간 뒤 아카이브 계층으로 이동
- 보존 기간이 지나면 자동 삭제
- 개인정보 보존 정책에 따라 일정 기간 후 삭제

이 기능을 쓰면 별도 삭제 스크립트를 만들지 않아도 비용 절감과 데이터 보존·파기 정책을 자동화할 수 있습니다.

## 13.5 AWS Lake Formation 4:25:08–4:26:51

AWS Lake Formation은 S3 기반 데이터 레이크를 **등록하고, 카탈로그화하고, 권한을 세밀하게 통제**하는 서비스입니다.

강의는 Lake Formation을 “안전한 데이터 레이크를 만들 때 반복되는 설정을 줄이는 서비스”로 설명합니다. 데이터 수집·저장·카탈로그·변환도 연결되지만, 시험에서 가장 중요한 역할은 **데이터 접근 제어와 거버넌스**입니다.

강의의 예시는 여러 부서가 각자 데이터 레이크를 운영하는 조직입니다.

```text
제조 현장 데이터 → S3
물류 데이터      → S3
급여 데이터      → S3
                  ↓
          Glue Data Catalog
                  ↓
          Lake Formation 권한 통제
                  ↓
DataZone 또는 SageMaker Unified Studio에서 검색·접근 요청
                  ↓
데이터 소유자 승인 후 분석 도구에서 사용
```

### 강의가 말하는 핵심

- S3에는 실제 데이터가 저장됨
- Glue Data Catalog에는 데이터의 위치·스키마 같은 메타데이터가 등록됨
- Lake Formation은 데이터베이스·테이블·컬럼·행 같은 단위로 접근을 통제함
- 사용자나 역할은 IAM 권한만 있어도 충분하지 않을 수 있음
- 데이터 검색, 발행, 구독, 접근 요청 같은 조직적 흐름에는 DataZone 계열 거버넌스 도구를 함께 쓸 수 있음

> **[용어 해설] Data Catalog와 Lake Formation의 차이**
>
> - **Glue Data Catalog**는 도서관의 카드 목록임. “이 데이터는 어디 있고, 어떤 컬럼과 타입을 가지는가”를 적어 둠.
> - **Lake Formation**은 도서관 출입·열람 권한 시스템임. “누가 어떤 테이블과 컬럼을 볼 수 있는가”를 통제함.
> - 실제 책, 즉 데이터 파일은 보통 S3에 있음. Catalog나 Lake Formation으로 데이터 파일 자체가 옮겨가는 것은 아님.

> **[2026 최신화]** 현재 Lake Formation은 Data Catalog 리소스와 S3 위치 및 실제 데이터에 대한 접근을 통제함. 요청은 IAM과 Lake Formation 양쪽 권한 검사를 통과해야 할 수 있음. 또한 컬럼·행·셀 단위의 세밀한 제어가 가능함. 따라서 시험에서 `S3 데이터 레이크`, `세밀한 데이터 권한`, `Athena/Glue/EMR/Redshift Spectrum에서 일관된 접근 통제`가 함께 나오면 Lake Formation이 정답 후보임. [공식 근거 6][6]

> **[시험 연결]** 이 강의가 IAM만으로 권한을 통제하는 인상을 줄 수 있으나, 최신 시험은 역할 기반 RBAC, 태그 기반 TBAC, 속성 기반 ABAC와 최소 권한, 데이터 주권까지 요구함. Lake Formation은 IAM의 대체재가 아니라 데이터 레이크 계층에 세밀한 권한을 추가하는 도구로 이해해야 함. [공식 근거 1][1]

# 14. 데이터베이스, 마이그레이션, 오프라인 전송

## 14.1 Amazon RDS와 Amazon Aurora 4:26:52–4:27:28

Amazon RDS Relational Database Service는 관계형 데이터베이스를 AWS에서 운영하기 쉽게 만든 관리형 서비스입니다. 백업, 패치, 장애 조치, 확장 같은 운영 부담을 AWS가 일부 대신 처리합니다.

강의는 RDS에서 사용할 수 있는 대표 엔진을 나열합니다.

- MySQL
- PostgreSQL
- MariaDB
- Oracle Database
- Microsoft SQL Server
- Amazon Aurora MySQL 호환
- Amazon Aurora PostgreSQL 호환

> **[용어 해설] RDS와 Aurora는 같은 급의 이름이 아님**
>
> - **RDS**는 관계형 데이터베이스를 관리해 주는 서비스 범주임.
> - **Aurora**는 MySQL·PostgreSQL과 호환되는 AWS의 관리형 관계형 데이터베이스 엔진임.
> - 따라서 “Aurora가 RDS와 완전히 별개”라고 보면 틀림. Aurora는 RDS 생태계 안에서 선택하는 엔진으로 이해하는 편이 맞음.

> **[2026 최신화]** Aurora는 MySQL 및 PostgreSQL과 호환되는 완전관리형 관계형 데이터베이스 엔진임. 최신 DEA-C01 시험 가이드 v1.1에서는 Aurora가 명시적으로 in-scope 서비스에 추가됨. `운영 트랜잭션`, `관계형 데이터`, `MySQL/PostgreSQL 호환`, `관리형 고가용성` 같은 요구에서 RDS·Aurora를 비교해야 함. [공식 근거 1][1] [공식 근거 12][12]

### 데이터 엔지니어링에서의 위치

RDS나 Aurora는 주로 운영계 OLTP 데이터의 원천입니다. 즉 주문, 결제, 회원, 서비스 트랜잭션처럼 애플리케이션이 실시간으로 기록·조회하는 데이터를 보관합니다.

분석용 대규모 집계는 보통 Redshift나 Athena 계층으로 넘깁니다.

```text
웹·앱 트랜잭션
  → RDS / Aurora
  → DMS CDC 또는 Zero-ETL Integration
  → Redshift / S3 Data Lake
  → Athena / Quick Sight 대시보드
```

> **[용어 해설] OLTP와 OLAP**
>
> - **OLTP Online Transaction Processing**: 주문 1건 등록, 잔액 수정처럼 짧고 정확한 운영 트랜잭션 처리
> - **OLAP Online Analytical Processing**: 수개월 매출 집계, 고객 행동 분석처럼 대량 데이터를 읽고 집계하는 분석 처리
>
> RDS·Aurora는 대체로 OLTP, Redshift는 OLAP에 더 맞음. 운영 DB에 무거운 분석 쿼리를 직접 던지는 설계는 성능·가용성 리스크를 만든다는 뜻임.

## 14.2 Amazon Timestream 4:27:29–4:27:59

Amazon Timestream은 시간 순서가 핵심인 데이터, 즉 시계열 데이터를 저장·분석하는 관리형 데이터베이스 계열입니다.

예시는 다음과 같습니다.

- 서버 CPU·메모리·지연 시간 모니터링
- IoT 센서 측정값
- 산업 장비 텔레메트리
- 실시간 관측 데이터
- 시간 단위 지표 분석

> **[용어 해설] 시계열 데이터**
>
> 값 하나만 보는 것이 아니라 “언제 발생했는가”가 데이터 의미의 일부인 데이터임.
>
> ```text
> 10:00 CPU 31%
> 10:01 CPU 87%
> 10:02 CPU 95%
> ```
>
> 위 데이터는 숫자만 보면 의미가 약하지만, 시간이 붙으면 급격한 부하 증가를 알 수 있음.

> **[시험 우선순위]** Timestream은 강의에서 소개하지만, 현재 DEA-C01 in-scope 서비스 목록에 명시된 핵심 축은 아님. 시계열·IoT·관측 데이터를 만나면 후보로 떠올리되, Glue·Athena·Redshift·Kinesis·S3·IAM·Lake Formation보다 우선순위를 높게 둘 필요는 없음. [공식 근거 1][1]

## 14.3 Amazon Neptune 4:28:00–4:28:22

Amazon Neptune은 관계 자체가 중요한 데이터를 다루는 관리형 그래프 데이터베이스 서비스입니다.

관계형 테이블은 행과 열 중심으로 데이터를 저장합니다. 그래프 데이터베이스는 노드와 관계선으로 데이터의 연결 구조를 잘 표현합니다.

```text
사용자 A ── 구매 ── 상품 X
사용자 A ── 팔로우 ── 사용자 B
사용자 B ── 구매 ── 상품 Y
```

이런 구조에서 “A와 두 단계 안에서 연결된 사용자”, “사기 의심 계정과 같은 결제수단을 공유한 계정” 같은 질의가 중요하면 그래프 모델이 적합할 수 있습니다.

> **[용어 해설] graph**
>
> - **Node**: 사람, 상품, 계정처럼 개체 하나
> - **Edge**: 구매함, 친구임, 연결됨처럼 개체 간 관계
>
> `관계의 탐색`이 핵심 요구라면 Neptune을 떠올리면 됨.

## 14.4 데이터베이스와 데이터 엔지니어링 아키텍처 4:28:23–4:30:47

강의는 데이터베이스를 DBA 관점으로 깊게 파기보다, 데이터 엔지니어링 흐름 안에서 원천·복제·분석 저장소로 연결하는 관점이 중요하다고 설명합니다.

### Aurora에서 Redshift로 분석 데이터 전달

운영 애플리케이션의 트랜잭션은 Aurora에 쌓입니다. 분석팀은 운영 DB를 직접 무겁게 조회하지 않고, 변경 데이터를 Redshift로 반영해 대시보드를 만듭니다.

```text
애플리케이션
  → Aurora 또는 RDS
  → Zero-ETL Integration 또는 DMS
  → Redshift
  → Quick Sight
```

### Blue/Green 배포 개념

강의는 기존 데이터베이스 Blue 환경에서 새 Green 환경으로 복제한 뒤, 검증이 끝나면 트래픽을 새 환경으로 전환하는 예시를 보여 줍니다.

> **[용어 해설] Blue/Green**
>
> - **Blue**: 현재 운영 중인 기존 환경
> - **Green**: 새 버전 또는 새 구성의 대체 환경
> - 먼저 Green에서 데이터·기능을 검증
> - 전환 시점에 트래픽을 Green으로 이동
>
> 핵심 목적은 업그레이드나 변경 중 서비스 중단과 롤백 위험을 줄이는 것임.

### DynamoDB 변경을 분석계로 넘기는 예시

DynamoDB에 변경이 생기면 Streams가 변경 이벤트를 기록하고, Lambda나 Kinesis를 거쳐 Redshift 같은 분석 저장소에 반영할 수 있습니다.

```text
DynamoDB
  → DynamoDB Streams
  → Lambda 또는 Kinesis
  → Redshift
  → Quick Sight
```

> **[주의]** 이것은 가능한 예시 아키텍처이지 DynamoDB 데이터를 Redshift로 보내는 유일한 정답은 아님. 변경량, 지연 허용치, 데이터 모델, 재처리 필요성, 운영 복잡도에 따라 DMS, Glue, Firehose, Kinesis, S3 데이터 레이크 패턴을 비교해야 함.

## 14.5 AWS Application Discovery Service 4:30:48–4:31:35

AWS Application Discovery Service는 온프레미스 데이터센터의 서버, 애플리케이션, 의존 관계, 자원 사용량 정보를 수집해 AWS 이전 계획을 세우는 도구입니다.

강의가 강조하는 포인트는 다음과 같습니다.

- 기존 서버와 애플리케이션을 조사
- 어떤 애플리케이션이 어떤 DB·서버·네트워크에 의존하는지 파악
- agent 기반 또는 agentless 방식으로 수집
- AWS Migration Hub, DMS 등 마이그레이션 도구와 연결

> **[용어 해설] dependency mapping**
>
> 시스템을 옮기기 전에 “이 서버가 어떤 데이터베이스에 연결되는가”, “이 애플리케이션이 어떤 API와 파일 공유를 사용하는가”를 지도처럼 그리는 작업임.
>
> 이 지도를 만들지 않으면 DB만 옮기고 실제 호출 애플리케이션이 끊기는 식의 실패가 생김.

## 14.6 AWS Database Migration Service AWS DMS 4:31:36–4:33:07

AWS DMS는 데이터베이스를 AWS로 옮기거나, 서로 다른 데이터 저장소 사이에 데이터를 복제할 때 쓰는 관리형 서비스입니다.

강의는 두 가지 마이그레이션을 구분합니다.

- **Homogeneous migration 동종 마이그레이션**: 같은 엔진 계열로 이동  
  예: 온프레미스 PostgreSQL → Amazon RDS for PostgreSQL
- **Heterogeneous migration 이기종 마이그레이션**: 다른 엔진 계열로 이동  
  예: PostgreSQL → MySQL

DMS에서는 보통 처음 전체 데이터를 옮긴 뒤, 이후 바뀌는 데이터만 지속적으로 따라가게 할 수 있습니다.

```text
기존 DB
  ├─ Full load: 현재 데이터 전체 복사
  └─ CDC: 이후 INSERT / UPDATE / DELETE 변경을 지속 반영
        ↓
목적지 RDS / Aurora / Redshift / S3 등
```

> **[용어 해설] CDC Change Data Capture**
>
> 전체 테이블을 매번 복사하지 않고, 마지막 처리 이후에 **새로 삽입·수정·삭제된 변경분만** 잡아내서 전달하는 방식임.
>
> 비유하면 “책 전체를 매일 다시 복사”하는 대신 “오늘 바뀐 페이지에 붙은 수정 스티커만 전달”하는 방식임.

> **[2026 최신화]** DMS는 현재도 Full load와 CDC 기반의 지속 복제를 지원하며, S3를 대상지로 쓸 때 CSV뿐 아니라 Parquet 저장을 선택할 수 있음. 반면 AWS Schema Conversion Tool AWS SCT는 DEA-C01 v1.1에서 in-scope 목록에서 빠졌음. 즉 시험에서 **DMS와 CDC는 중요**, SCT 세부 기능은 우선순위를 낮춰도 됨. [공식 근거 1][1] [공식 근거 7][7]

## 14.7 AWS Snow Family 4:33:08–4:34:28

AWS Snow Family는 인터넷으로 옮기기 힘들 정도로 큰 데이터를 물리 장비에 담아 AWS로 전송하는 서비스군입니다.

강의의 요지는 단순합니다.

```text
인터넷 회선 전송이 비현실적임
  → AWS가 물리 장비를 보냄
  → 고객이 데이터를 적재
  → 장비를 AWS로 반송
  → AWS가 지정한 S3 위치 등으로 가져감
```

사용 사례는 다음과 같습니다.

- 수십 TB 이상 대규모 초기 이전
- 회선이 느리거나 불안정한 지역
- 재난·현장·격리망 환경
- 엣지 환경에서 일부 처리 후 데이터 반송

강의는 Snowcone과 Snowball Edge의 용량·컴퓨팅 옵션을 언급합니다.

> **[원문 주의]** 물리 장비 모델과 제공 용량은 시점·리전·제품 수명주기에 따라 변할 수 있음. 단기 시험 준비에서 `8TB`, `80TB` 같은 숫자 암기에 시간을 버리지 말 것. **네트워크로 옮기기 어려운 대용량 데이터의 오프라인 전송**이라는 선택 기준이 핵심임.

# 15. 애플리케이션 통합 서비스

## 15.1 Amazon SNS 4:34:29–4:34:50

Amazon SNS Simple Notification Service는 발행자와 여러 구독자를 연결하는 pub/sub 메시징 서비스입니다.

하나의 이벤트가 발생했을 때 여러 대상에게 알림이나 메시지를 보내고 싶을 때 사용합니다.

```text
Glue Job 실패
  → SNS Topic
  ├─ 이메일 알림
  ├─ Lambda 실행
  └─ SQS Queue 전달
```

> **[용어 해설] Topic과 Pub/Sub**
>
> - **Topic**: 메시지를 발행하는 주제 채널
> - **Publisher**: Topic에 메시지를 보내는 쪽
> - **Subscriber**: Topic을 구독해 메시지를 받는 쪽
>
> “한 사건을 여러 곳에 알린다”가 SNS의 핵심임.

## 15.2 Amazon SQS 4:34:51–4:35:54

Amazon SQS Simple Queue Service는 컴포넌트 사이에 메시지를 잠시 보관해 비동기 처리를 가능하게 만드는 큐 서비스입니다.

```text
Producer
  → SQS Queue
  → Consumer
```

강의는 SQS가 마이크로서비스나 분산 시스템을 느슨하게 결합하고, 트래픽 급증을 완충하고, 실패한 작업을 재시도하는 데 도움이 된다고 설명합니다.

예를 들어 Lambda가 어떤 작업을 하다가 실패하면, 메시지를 SQS에 보내고 별도 소비자가 다시 처리하게 만들 수 있습니다. 이때 DynamoDB 같은 저장소에 처리 상태를 기록할 수도 있습니다.

> **[용어 해설] SNS와 SQS 차이**
>
> - **SNS**: 한 이벤트를 여러 구독자에게 퍼뜨리는 방송
> - **SQS**: 한 작업 메시지를 안전하게 줄 세워 소비자에게 넘기는 대기열
>
> SNS는 fan-out, SQS는 decoupling과 retry buffer에 더 가깝다고 보면 됨.

> **[용어 해설] DLQ Dead-Letter Queue**
>
> 여러 번 재시도해도 처리하지 못한 메시지를 본 큐에서 분리해 보관하는 실패 격리소임. 재시도 무한 루프를 막고, 운영자가 실패 원인을 조사하게 해 줌.

## 15.3 Amazon AppFlow 4:35:55–4:37:01

Amazon AppFlow는 SaaS Software as a Service 애플리케이션과 AWS 서비스 사이에서 데이터를 안전하게 옮기는 관리형 통합 서비스입니다.

강의가 제시하는 흐름은 다음과 같습니다.

```text
Salesforce / SAP / Google Analytics 같은 SaaS
  → AppFlow
  → 필요 시 마스킹·필터링·매핑
  → S3 / Redshift 등 AWS 대상지
```

시험에서 다음 단서가 함께 나오면 AppFlow를 우선 검토합니다.

- 외부 SaaS 데이터
- Salesforce, 마케팅·분석 도구 등
- 코드를 많이 쓰지 않고 관리형으로 가져오고 싶음
- S3나 Redshift에 적재하려 함

> **[용어 해설] SaaS**
>
> 설치형 프로그램이 아니라 인터넷으로 구독해 쓰는 완성형 소프트웨어 서비스임. Salesforce, Google Analytics 같은 외부 서비스에서 데이터를 AWS로 옮기고 싶다는 요구가 나오면 AppFlow 맥락일 가능성이 큼.

## 15.4 Amazon EventBridge 4:37:02–4:38:14

Amazon EventBridge는 AWS 서비스, SaaS, 사용자 애플리케이션에서 발생한 이벤트를 받아 규칙에 따라 대상 서비스로 전달하는 서버리스 이벤트 버스입니다.

강의 예시는 Kafka 이벤트를 받아 필요하면 Lambda로 변환·보강한 뒤, 다른 Event Bus나 AWS 서비스로 전달하는 구조입니다.

```text
Kafka / AWS Service / SaaS 이벤트
  → EventBridge
  → Rule
  → Lambda로 변환·보강
  → Target Event Bus / SQS / Step Functions / 기타 서비스
```

> **[용어 해설] Event와 Message**
>
> - **Event**: “무슨 일이 일어났다”는 사실. 예: `파일 업로드 완료`, `주문 생성`, `Glue Job 실패`
> - **Message**: 전달할 데이터 묶음
>
> EventBridge는 사건 기반 라우팅에 강점이 있고, SQS는 작업 메시지 대기열에 강점이 있음.

> **[용어 해설] EventBridge Pipes**
>
> Source에서 메시지를 받고, 필요하면 필터링·보강을 거친 후 Target으로 보내는 연결 파이프임.
>
> ```text
> Source → Filter → Enrichment Lambda → Target
> ```

# 17. 최신 DEA-C01 보완 범위: 이 강의에 거의 없는 영역

## 17.1 Apache Iceberg와 Open Table Format

Apache Iceberg는 데이터 레이크에서 테이블을 안정적으로 관리하기 위한 오픈 테이블 형식입니다.

기존 S3 + Parquet만 쓰는 구조는 파일은 저장하기 쉽지만, 다음 문제가 생깁니다.

- 여러 작업이 동시에 테이블을 수정할 때 일관성 관리가 어려움
- 스키마 변경 이력 관리가 불편함
- 어떤 파일이 현재 테이블에 속하는지 관리가 복잡함
- 시간 여행, 스냅샷, 원자적 커밋 같은 기능이 필요해짐

Iceberg는 이런 테이블 메타데이터와 스냅샷을 관리합니다.

```text
S3의 Parquet 파일들
  + Iceberg 메타데이터
  = 하나의 신뢰 가능한 분석 테이블
```

> **[용어 해설] Open table format**
>
> 데이터 파일 형식 Parquet 자체가 아니라, 그 파일들을 “하나의 테이블”로 다루게 해 주는 규약임.
>
> Parquet이 종이라면 Iceberg는 목차, 버전 관리, 변경 이력, 파일 목록을 맡는 관리 체계에 가까움.

> **[2026 최신화]** DEA-C01 v1.1에 `open table format 관리`가 새 기술 항목으로 추가됨. AWS Glue, EMR, Athena, Redshift는 Iceberg를 지원하며, Amazon S3 Tables는 Iceberg 테이블을 위한 별도 table bucket을 제공함. [공식 근거 1][1] [공식 근거 8][8]

## 17.2 Amazon S3 Tables

Amazon S3 Tables는 Apache Iceberg 형식의 테이블을 S3에서 관리하기 쉽게 만든 관리형 테이블 스토리지 기능입니다.

기존 S3 버킷과 달리 `table bucket`이라는 별도 버킷 유형에서 테이블을 관리하며, Iceberg를 지원하는 Athena, Redshift, Spark 등의 엔진으로 SQL 질의를 할 수 있습니다.

> **[시험 연결]** `대규모 표 형식 데이터`, `S3 기반 분석`, `Iceberg`, `테이블 유지관리 부담 감소`, `Athena/Redshift/Spark 호환`이라는 단서가 같이 나오면 S3 Tables를 검토함.

## 17.3 벡터, HNSW, IVF

최신 시험은 일반적인 데이터 엔지니어링에 벡터와 벡터 인덱스 기초를 추가했습니다.

- **Vector 벡터**: 텍스트·이미지 같은 데이터를 숫자 배열로 표현한 것
- **Embedding 임베딩**: 의미가 비슷한 데이터가 수학적으로 가까운 벡터가 되게 변환한 표현
- **Vector index 벡터 인덱스**: 수많은 벡터 중 비슷한 벡터를 빠르게 찾기 위한 인덱스
- **HNSW**: 근접 이웃 탐색을 빠르게 하기 위한 그래프 기반 인덱스 계열
- **IVF**: 벡터를 여러 그룹으로 나눈 뒤 관련 그룹부터 탐색하는 인덱스 계열

> **[시험 수준]** 알고리즘을 구현할 수준까지 갈 필요 없음. “의미 유사도 검색을 빠르게 하려면 벡터 인덱스가 필요하며, HNSW·IVF는 그 후보”라는 수준과, Aurora PostgreSQL 등 적절한 저장소 선택 맥락을 잡으면 됨. [공식 근거 1][1]

## 17.4 LLM을 활용한 데이터 처리

최신 시험 가이드에는 `LLM을 데이터 처리에 통합`하는 기술 항목이 추가되었습니다.

예시는 다음과 같습니다.

- 자유 텍스트에서 구조화 필드 추출
- 대량 문서 요약·분류
- 데이터 품질 이상 설명 생성
- 자연어 질의를 SQL 초안으로 변환
- 비정형 데이터의 태그·메타데이터 생성

> **[주의]** LLM은 정답 보장 엔진이 아님. 데이터 파이프라인에 넣을 때는 결과 검증, 재시도, 원본 보존, 프롬프트·모델 버전 기록, PII 통제를 같이 설계해야 함.
>
> **시험 관점**: “LLM이 필요하다”는 사실만으로 모든 데이터를 LLM에 보내는 선택지는 보통 과함. 대량 처리 비용, 보안, 정확도, 비동기 처리, 사람 검토 필요성을 함께 판단해야 함.

## 문제의 핵심

규제가 강한 회사가 Kinesis Data Streams로 스트리밍 솔루션을 만들려고 합니다.

요구사항은 다음과 같습니다.

- 스트림에 기록되는 데이터는 암호화되어야 함
- 키 회전과 키 정책을 조직이 통제해야 함

이 상황에서 답의 핵심은 Kinesis Data Streams에 서버 측 암호화를 설정하고, AWS KMS의 고객 관리형 키를 사용한다는 것입니다.

```text
Producer
  → Kinesis Data Streams
  → KMS key로 저장 시 암호화 at rest
```

> **[용어 해설] At rest와 In transit**
>
> - **At rest**: 저장되어 있는 동안의 암호화. 예: Kinesis 스트림 저장 계층, S3 객체, Redshift 스냅샷
> - **In transit**: 네트워크로 이동하는 동안의 암호화. 보통 TLS
>
> 둘은 별개 요구임. 저장 암호화만 켜도 전송 구간 보안이 자동으로 설명되는 것은 아님.

> **[원문 정정 및 2026 최신화]** 원문의 `Customer Master Key CMK`라는 표현은 현재 `KMS key` 또는 `customer managed key`로 바뀌는 중임. 개념은 같지만, 고객이 키 정책·권한·회전 등을 통제해야 한다면 **customer managed key**가 적절한 표현임. Kinesis Data Streams의 서버 측 암호화는 KMS 키를 이용해 저장 전 데이터를 암호화함. [공식 근거 10][10]

## 오답 제거 방식

- “Kinesis는 자동 암호화되므로 아무 조치도 필요 없다”  
  → 고객의 키 통제 요구를 만족하는 설명이 부족함.

- “해시만 해서 보내면 된다”  
  → 해시는 원문 복원이 안 되는 단방향 변환임. 저장 암호화와 키 회전 요구의 답이 아님.

- “스트림 생성 중 암호화만 켜면 된다”  
  → 문제에서 고객이 키 회전·정책을 직접 통제해야 한다고 했으므로, 어떤 KMS 키를 선택할지까지 봐야 함.

> **[시험 풀이 규칙]** 보안 문제는 `암호화 필요`만 보지 말고 다음을 분해해야 함.
>
> 1. 저장 중인가 전송 중인가  
> 2. AWS가 관리하는 키면 되는가, 고객이 키를 통제해야 하는가  
> 3. IAM 권한, KMS Key Policy, 서비스 역할 중 어디가 병목인가  
> 4. 감사 로그·키 회전·최소 권한 요구가 있는가

# 19. 강의 종료 후 학습 판단

이 강의는 AWS 데이터 엔지니어링 서비스 지형을 한 번에 보여 주는 데 쓸모가 있습니다. 특히 아래 흐름을 잡는 데 적합합니다.

```text
수집
  → S3 / Kinesis / DMS / AppFlow
변환
  → Glue / EMR / Lambda / Flink
저장·분석
  → S3 Data Lake / Athena / Redshift / DynamoDB
오케스트레이션
  → Glue Trigger / Step Functions / EventBridge / MWAA
보안·거버넌스
  → IAM / KMS / Lake Formation / CloudTrail / Macie
제공
  → Quick Sight / 데이터 공유 / Catalog
```

다만 2024년 강의라는 한계가 분명합니다. 이 문서의 최신화 주석까지 함께 본 뒤, 시험 대비에서는 아래 순서로 이해를 굳히는 편이 맞습니다.

1. S3, Glue Data Catalog, Glue ETL, Athena
2. Redshift, Parquet, partition, Iceberg, S3 Tables
3. Kinesis Data Streams, Firehose, MSK, Lambda, EventBridge
4. DMS CDC, RDS·Aurora, DynamoDB
5. IAM, KMS, Secrets Manager, Lake Formation, Macie, CloudTrail, AWS Config
6. SageMaker Unified Studio·Catalog, LLM, vector 개념
7. SQL, 재처리, idempotency, 오류 격리, 모니터링, 비용 선택 기준
