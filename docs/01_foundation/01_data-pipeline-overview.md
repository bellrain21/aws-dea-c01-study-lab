# 1. 이 문서의 목적

DEA-C01은 AWS 서비스 이름을 많이 외우는 시험이 아님.

문제는 보통 아래처럼 나옴.

```text
어떤 데이터가
어디서 들어오고
얼마나 빨리 처리되어야 하며
어디에 저장되고
누가 조회하며
실패했을 때 어떻게 다시 처리되어야 하는가
```

이 질문에 맞춰 서비스를 고르는 시험임.

따라서 지금 이 문서의 목표는 서비스 이름을 외우는 것이 아니라 아래 흐름을 이해하는 것임.

```text
데이터 원천
→ 수집
→ 저장
→ 변환
→ 검증
→ 제공
→ 감시
```

---

# 2. 데이터 엔지니어링이란 무엇인가

## 2.1 한 줄 정의

```text
데이터 엔지니어링은 데이터가 안전하게 들어오고,
쓸 수 있는 형태로 바뀌고,
필요한 사람이 다시 사용할 수 있게 만드는 흐름을 설계·운영하는 일임.
```

데이터 분석가가 “이 데이터에서 무엇을 해석할까”에 더 가깝다면,  
데이터 엔지니어는 “이 데이터가 믿을 만한 상태로 계속 도착하게 하려면 무엇이 필요한가”에 가까움.

| 역할 | 핵심 질문 |
|---|---|
| 데이터 엔지니어 | 데이터가 어떻게 안정적으로 들어오고, 변환되고, 제공되는가 |
| 데이터 분석가 | 데이터가 무엇을 말하는가 |
| 백엔드 엔지니어 | 사용자 요청을 처리하는 애플리케이션이 어떻게 동작하는가 |
| 데이터 사이언티스트·ML 엔지니어 | 데이터를 이용해 예측·분류·추천을 어떻게 만드는가 |

실무에서는 경계가 겹치지만, DEA-C01은 데이터 엔지니어 관점이 중심임.

# 3. 데이터 파이프라인

## 3.1 용어 정의

| 영문 | 발음 | 한국어 | 실제 의미 |
|---|---|---|---|
| Data pipeline | 데이터 파이프라인 | 데이터 처리 흐름 | 원천에서 목적지까지 데이터를 옮기고 바꾸고 검증하는 전체 과정 |
| Source | 소스 | 원천 | 데이터가 처음 생기는 곳 |
| Data source | 데이터 소스 | 데이터 원천 | DB, API, CSV, 로그, 이벤트 등 |
| Producer | 프로듀서 | 생산자 | 데이터를 보내는 시스템 또는 주체 |
| Ingestion | 인제스천 | 수집·유입 | 원천 데이터를 처리 경로 안으로 가져오는 단계 |
| Transform | 트랜스폼 | 변환 | 데이터 형식·값·구조를 바꾸는 단계 |
| Target / Sink | 타깃 / 싱크 | 목적지 | 변환 결과가 도착하는 저장소 또는 서비스 |
| Consumer | 컨슈머 | 소비자 | 데이터를 사용하는 분석가, 대시보드, 애플리케이션, 모델 등 |
| Dataset | 데이터셋 | 데이터 집합 | 함께 관리·분석하는 데이터 묶음 |

---

## 3.2 최소 흐름

```text
주문 서비스 DB 또는 외부 API
→ 수집
→ 원본 저장
→ 정제·형변환·중복 제거
→ 품질 검증
→ 분석용 데이터 제공
→ 대시보드·리포트·모델이 사용
```

이 흐름을 공장으로 보면:

```text
원천 데이터 = 입고된 원재료
수집 = 원재료를 공장 안으로 반입
저장 = 원재료와 중간 결과를 보관
변환 = 세척·분류·가공
검증 = 불량 검사
제공 = 완성품 출고
감시 = 공정 장애 확인
```

비유는 구조를 이해하기 위한 것임.  
데이터는 물리적 원재료가 아니므로 복제·재처리·버전 관리가 가능하다는 차이는 있음.

# 4. 각 단계는 왜 필요한가

| 단계 | 하는 일 | 없으면 생기는 문제 |
|---|---|---|
| 수집 | DB, API, 로그, 파일에서 데이터를 가져옴 | 분석 대상이 들어오지 않음 |
| 저장 | 원본과 가공 결과를 보관 | 재처리·감사·분석 근거가 사라짐 |
| 변환 | 타입 정리, 중복 제거, 조인, 집계, 마스킹 | 데이터 형식이 제각각이라 함께 쓰기 어려움 |
| 검증 | 누락, 중복, 범위 오류, 타입 오류를 확인 | 잘못된 데이터가 분석·대시보드로 전달됨 |
| 제공 | 분석·제품·모델이 읽기 좋게 준비 | 소비자가 원천을 매번 직접 처리해야 함 |
| 감시 | 실패, 지연, 비용, 품질 이상을 확인 | 파이프라인이 멈춰도 늦게 발견함 |

## 4.1 예시: 일별 주문 매출

```text
1. 주문 DB에서 어제 주문 데이터를 가져옴
2. 원본 파일을 저장함
3. 금액 문자열을 숫자로 바꿈
4. 중복 주문을 제거함
5. 주문 건수와 금액 범위를 검증함
6. 일별 매출 집계 데이터를 만듦
7. BI 대시보드가 집계 데이터를 읽음
8. 실패하면 알림을 보내고 재실행함
```

여기서 중요한 것은 “어떤 서비스 이름을 쓰는가”가 아님.

먼저 아래가 정해져야 함.

```text
입력은 무엇인가
언제 들어오는가
얼마나 늦어도 되는가
어떤 규칙으로 변환하는가
누가 어떤 형태로 쓰는가
실패하면 어떻게 복구하는가
```

서비스는 그 다음에 고름.

# 5. ETL과 ELT

## 5.1 정의

| 용어 | 발음 | 풀네임 | 순서 |
|---|---|---|---|
| ETL | 이티엘 | Extract, Transform, Load | 꺼냄 → 바꿈 → 적재 |
| ELT | 이엘티 | Extract, Load, Transform | 꺼냄 → 적재 → 저장소 내부에서 변환 |

```text
ETL
원본
→ 변환 엔진
→ 분석 저장소

ELT
원본
→ 분석 저장소
→ 저장소 내부 SQL·변환
```

## 5.2 차이는 무엇인가

| 관점 | ETL | ELT |
|---|---|---|
| 변환 시점 | 목적지에 넣기 전 | 목적지에 넣은 뒤 |
| 장점 | 목적지에 정제된 데이터만 넣기 쉬움 | 원본을 먼저 확보하고 변환을 유연하게 바꾸기 쉬움 |
| 주의점 | 변환 실패 시 목적지에 데이터가 늦게 도착할 수 있음 | 저장소 비용·처리 비용·권한 관리 고려 필요 |
| 시험 관점 | 둘 중 하나를 외우지 말고 요구사항으로 선택 | 둘 중 하나를 외우지 말고 요구사항으로 선택 |

### 오해 제거

```text
ETL이 항상 더 안전한 것도 아니고,
ELT가 항상 더 최신 방식인 것도 아님.
```

선택 기준은 데이터 규모, 원본 품질, 변환량, 저장소 성능, 비용, 재처리 요구사항임.

# 6. 파이프라인에서 반드시 생각할 네 가지

## 6.1 정확성 Correctness

```text
결과가 실제 데이터와 규칙에 맞는가
```

예:

```text
매출 금액이 음수가 아닌가
주문 ID가 중복되지 않았는가
날짜가 올바른 형식인가
```

## 6.2 신뢰성 Reliability

```text
실패해도 다시 실행되어 정상 결과를 만들 수 있는가
```

예:

```text
네트워크 오류로 작업이 중단됨
→ 재시도
→ 같은 주문이 두 번 적재되면 안 됨
```

## 6.3 비용 Cost

```text
필요 이상으로 저장·처리·조회 비용을 쓰지 않는가
```

예:

```text
하루치만 분석하면 되는데
5년치 파일 전체를 읽으면 비용과 시간이 증가함
```

## 6.4 보안 Security

```text
누가 어떤 데이터에 접근할 수 있는가
민감정보가 노출되지 않는가
```

예:

```text
분석가는 집계 데이터만 조회
운영자는 원본 접근 가능
개인정보 컬럼은 제한된 역할만 조회
```

# 7. 문제를 읽는 순서

DEA-C01 문제를 읽을 때 서비스 이름부터 찾지 않음.

아래 순서로 읽음.

```text
1. 입력 데이터는 무엇인가
2. 배치인가, 실시간인가
3. 저장·조회·변환 중 어디가 핵심인가
4. 비용·지연·운영 부담 중 무엇이 우선인가
5. 보안·재처리·감사 요구가 있는가
6. 그 조건에 맞는 AWS 서비스를 고름
```

예시:

```text
조건:
매일 새벽 전날 주문 파일을 처리
낮은 비용 우선
수 분 지연 허용
실패 시 재실행 필요

판단:
실시간 처리보다 배치 처리 우선 검토
원본 보관과 재처리 구조 필요
```

이 시점에서는 서비스명을 확정하지 않아도 됨.  
`왜 배치인가`, `왜 원본 보관이 필요한가`를 판단하는 것이 먼저임.

# 10. 최종 1분 답안

```text
데이터 파이프라인은 원천 데이터가 분석·제품·모델에서 쓸 수 있는 데이터가 되기까지의 처리 흐름임.

데이터 엔지니어링은 데이터를 수집하고 저장하고 변환하고 검증해,
필요한 사람이 안전하게 다시 사용할 수 있게 만드는 일임.

ETL은 변환 후 적재하고,
ELT는 먼저 적재한 뒤 저장소 내부에서 변환하는 방식임.
둘 중 무엇을 쓸지는 데이터 규모, 비용, 원본 품질, 재처리 요구사항으로 결정함.
```
