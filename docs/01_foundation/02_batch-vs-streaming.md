# 1. 이 문서의 목적

데이터가 들어온다고 해서 항상 즉시 처리해야 하는 것은 아님.

```text
매일 새벽 전날 주문을 정리하는 일
≠
사용자 클릭이 발생한 뒤 수 초 안에 반영해야 하는 일
```

이 차이를 구분하지 못하면 AWS 서비스를 골라도 이유가 없음.

이 문서의 목표는 아래 질문에 답하는 것임.

```text
이 데이터는 지금 처리해야 하는가
아니면 모아서 처리해도 되는가
지연을 얼마나 허용하는가
재처리와 비용은 얼마나 중요한가
```

---

# 2. 핵심 용어

| 영문 | 발음 | 한국어 | 실제 의미 |
|---|---|---|---|
| Batch | 배치 | 일괄 처리 | 데이터를 일정 시간·수량만큼 모아 한 번에 처리 |
| Streaming | 스트리밍 | 연속 처리 | 이벤트가 들어오는 흐름을 지속적으로 처리 |
| Near real time | 니어 리얼타임 | 준실시간 | 즉시는 아니지만 수 초~수 분 정도의 짧은 지연 허용 |
| Event | 이벤트 | 발생 사실 | 클릭, 결제, 주문 생성, 센서 측정처럼 어떤 일이 발생했다는 기록 |
| Latency | 레이턴시 | 지연 시간 | 데이터가 발생한 시점부터 사용 가능해질 때까지 걸린 시간 |
| Throughput | 스루풋 | 처리량 | 단위 시간당 처리하는 데이터 양 |
| Burst | 버스트 | 순간 폭증 | 짧은 시간에 데이터가 급격히 몰리는 현상 |
| Consumer | 컨슈머 | 소비자 | 스트림이나 결과 데이터를 읽어 쓰는 주체 |

# 3. 배치 처리 Batch

## 3.1 정의

```text
배치는 데이터를 모아서,
정해진 시간 또는 정해진 양 단위로 처리하는 방식임.
```

예:

```text
매일 02:00에 전날 주문 데이터를 처리
매주 월요일에 지난주 광고 성과를 집계
매월 말에 정산 데이터를 계산
파일이 10만 건 쌓이면 한 번에 변환
```

## 3.2 배치가 유리한 상황

| 조건 | 이유 |
|---|---|
| 수 분~수 시간 지연 허용 | 즉시 처리 인프라가 필요하지 않음 |
| 하루·주 단위 리포트 | 결과가 즉시 필요하지 않음 |
| 대량 데이터를 모아 처리 | 한 번에 처리해 운영을 단순화 가능 |
| 비용 효율 우선 | 연속 실행보다 자원을 필요할 때만 쓰기 쉬움 |
| 과거 기간 재처리 필요 | 날짜·파일 단위로 다시 돌리기 쉬움 |

## 3.3 배치의 약점

```text
실시간 반응이 어렵다.
처리 주기 전까지 새 데이터가 결과에 반영되지 않는다.
한 번의 대형 작업이 실패하면 복구 범위가 커질 수 있다.
```

# 4. 스트리밍 Streaming

## 4.1 정의

```text
스트리밍은 데이터가 계속 들어오는 흐름을
도착에 가깝게 지속적으로 처리하는 방식임.
```

예:

```text
사용자 클릭 로그
주식 체결 이벤트
IoT 센서 값
부정 거래 탐지
게임 접속·이탈 이벤트
```

## 4.2 스트리밍이 유리한 상황

| 조건 | 이유 |
|---|---|
| 수 초 이내 반영 필요 | 배치 대기 시간이 허용되지 않음 |
| 이벤트가 계속 발생 | 파일 단위보다 이벤트 흐름이 자연스러움 |
| 여러 소비자가 같은 이벤트를 사용 | 분석·알림·저장 등 여러 후속 처리 가능 |
| 즉시 이상 탐지 필요 | 늦게 발견하면 손실이 커질 수 있음 |
| 실시간 대시보드 필요 | 최신 상태를 빠르게 보여야 함 |

## 4.3 스트리밍의 약점

```text
구조가 더 복잡해질 수 있다.
순서 보장, 중복, 지연 도착, 재처리 기준을 더 신경 써야 한다.
연속 처리 자원과 운영 비용을 고려해야 한다.
```

# 5. 배치와 스트리밍 비교

| 기준 | 배치 | 스트리밍 |
|---|---|---|
| 처리 시점 | 일정 시간·수량 후 | 이벤트 도착에 가깝게 |
| 지연 시간 | 분·시간·일 단위 허용 가능 | 초·분 단위 요구 가능 |
| 데이터 형태 | 파일, 일별 덤프, 정기 추출 | 이벤트, 로그, 센서, 클릭 |
| 운영 복잡도 | 상대적으로 단순 | 상대적으로 복잡 |
| 비용 관점 | 지연 허용 시 효율적일 수 있음 | 낮은 지연 요구로 비용 증가 가능 |
| 재처리 | 날짜·파일 단위로 처리하기 쉬움 | 이벤트 보관·재생 정책 필요 |
| 대표 목적 | 정산, 일별 집계, 정기 리포트 | 실시간 알림, 이상 탐지, 실시간 지표 |

# 6. 준실시간은 중간 지점임

현실은 배치와 스트리밍 둘 중 하나로 깔끔하게 나뉘지 않음.

```text
완전 실시간:
수 초 이내

준실시간:
수 분 이내

배치:
정해진 주기마다 수십 분·수 시간·하루 단위
```

예시:

| 요구사항 | 적합한 처리 사고 |
|---|---|
| 부정 결제를 수 초 안에 차단 | 실시간에 가까운 스트리밍 |
| 5분마다 매출 대시보드 갱신 | 준실시간 |
| 매일 아침 전날 매출 보고서 생성 | 배치 |
| 월말 세금 정산 | 배치 |

### 오해 제거

```text
스트리밍이 배치보다 무조건 좋은 것이 아님.
필요한 지연 시간보다 더 빠른 구조를 만들면 비용·운영 복잡도만 증가할 수 있음.
```

# 7. DEA-C01 문제를 푸는 판단 순서

서비스 이름을 보기 전에 아래 5개를 확인함.

```text
1. 허용 지연 시간은 얼마인가
2. 데이터는 파일·DB 덤프인가, 연속 이벤트인가
3. 데이터가 순간적으로 몰리는가
4. 실패 후 과거 데이터를 다시 처리해야 하는가
5. 비용과 운영 단순성 중 무엇이 우선인가
```

## 7.1 판단 예시

### 사례 A. 일별 주문 집계

```text
조건:
매일 02:00에 전날 주문 데이터를 집계
최대 2시간 지연 허용
실시간 알림 불필요
```

판단:

```text
배치 우선.
실시간 스트리밍을 먼저 고를 이유가 약함.
```

### 사례 B. 카드 부정 사용 탐지

```text
조건:
결제 이벤트가 발생하면 수 초 안에 검사
의심 거래는 즉시 차단 또는 알림
```

판단:

```text
스트리밍 또는 실시간 이벤트 처리 우선.
하루 뒤 배치 처리하면 목적을 달성하지 못함.
```

### 사례 C. 5분 단위 운영 대시보드

```text
조건:
물류 현황을 5분마다 갱신
수 초 단위 정확도는 필요 없음
```

판단:

```text
준실시간.
완전 실시간 구조가 과한지 먼저 검토.
```

# 8. 지연 시간과 처리량은 다름

둘을 섞으면 안 됨.

| 구분 | 질문 | 예시 |
|---|---|---|
| Latency 지연 시간 | 한 건이 얼마나 빨리 결과에 반영되는가 | 주문 발생 후 3초 뒤 알림 |
| Throughput 처리량 | 1초에 얼마나 많은 데이터를 처리하는가 | 초당 10,000건 처리 |

예:

```text
초당 10,000건을 처리해도
각 이벤트가 10분 뒤 반영되면 지연 시간은 높을 수 있음.

반대로 초당 10건만 처리해도
1초 안에 반영되면 지연 시간은 낮을 수 있음.
```

# 9. 스트리밍에서 자주 생기는 문제

서비스 구현 전에 문제 자체를 먼저 이해함.

| 문제 | 왜 생김 | 기본 대응 사고 |
|---|---|---|
| 중복 이벤트 | 재전송, 재시도, 네트워크 장애 | 고유 ID, 중복 제거, 멱등 처리 |
| 순서 뒤바뀜 | 네트워크·분산 처리 지연 | 순서 보장 범위와 이벤트 시간 구분 |
| 늦게 도착한 이벤트 | 발생 시각과 도착 시각이 다름 | 허용 지연 범위와 수정 정책 정의 |
| 순간 폭증 | 이벤트가 짧은 시간에 몰림 | 버퍼·확장·처리량 설계 |
| 재처리 필요 | 변환 로직 변경 또는 오류 | 이벤트 보관·재생 가능성 검토 |

이 문서에서는 해결 서비스명을 외우지 않음.  
`왜 이런 문제가 생기는가`만 먼저 잡음.

# 10. 배치도 재시도가 필요함

배치는 단순하지만 실패가 없다는 뜻은 아님.

```text
어제 주문 파일 처리 중 네트워크 오류 발생
→ 같은 날짜 데이터를 재실행
→ 이미 적재된 주문이 중복되면 안 됨
```

따라서 배치에도 아래가 필요함.

```text
실행 단위 정의
처리 완료 기록
중복 방지
재시도
과거 기간 재처리 Backfill
```

`Backfill`은 과거 날짜·기간의 데이터를 다시 처리하는 것임.

예:

```text
변환 규칙을 수정함
→ 지난 30일 데이터에 새 규칙 적용
→ 과거 30일을 다시 처리
```

# 12. 최종 1분 답안

```text
배치는 데이터를 일정 시간이나 수량만큼 모아 처리하는 방식이고,
스트리밍은 이벤트가 들어오는 흐름을 도착에 가깝게 지속 처리하는 방식임.

선택 기준은 데이터 형식보다 허용 지연 시간, 처리량, 재처리 필요성,
비용, 운영 복잡도임.

실시간이 필요하지 않은데 스트리밍을 쓰면
복잡도와 비용만 늘어날 수 있음.
```
